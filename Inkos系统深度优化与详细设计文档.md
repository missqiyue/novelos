# Inkos 系统深度优化与详细设计文档

## 1. 背景与核心痛点分析

在长篇小说（百万字级）的自动化生成过程中，当前的 Inkos 架构虽然提供了优秀的基础设施（PipelineRunner、StateManager、ContinuityAuditor 和 Python RAG），但在实际运行中，依然暴露出以下三大核心痛点：

### 1.1 剧情遗忘与伏笔断裂 (Plot & Hook Amnesia)
*   **现象**：AI 经常挖坑不填，几十章前设定的目标、敌人或关键道具在后续剧情中彻底消失。
*   **归因**：
    1.  **RAG 检索是被动的**：当前的向量检索严重依赖当前章节的语义查询（Query）。如果当前章是日常互动，RAG 根本不会召回早期埋下的危机线索。
    2.  **上下文“Lost in the Middle”**：随着《故事链管理台账》和 `pending_hooks.md` 的膨胀，将全量表格塞给 LLM 会导致严重的注意力衰减，AI 无法在庞大的上下文中精准定位到“已滞后 10 章未推进”的伏笔。

### 1.2 角色 OOC (Out of Character) 与降智
*   **现象**：配角说话口吻同质化（“万能 AI 腔”），或做出违背其“核心信念/底线”的决策。
*   **归因**：
    1.  **角色上下文缺乏动态过滤**：在生成正文时，系统将所有人物的设定全量输入。出场角色的关键设定（三钉、口头禅）被大量未出场角色的信息淹没。
    2.  **单体 Auditor 认知过载**：目前的 `ContinuityAuditor` 试图用一个 Prompt 完成 37 个维度的审查（包括 OOC、战力、逻辑等），导致 LLM 变为“橡皮图章”，对细微的人设偏离失去敏感度。

### 1.3 状态幻觉与“死者苏生” (State Hallucinations)
*   **现象**：角色拿着已经碎掉的剑战斗，或者死去的配角突然复活。
*   **归因**：
    1.  **缺乏时序感知的纯语义检索**：向量检索擅长找“相关性”而非“最新状态”。搜索“反派A”，RAG 可能会召回 50 章前他大杀四方的片段，而忽略了他 10 章前已经被杀的事实，导致 AI 拿到过期上下文。
    2.  **Markdown 作为 Truth File 的脆弱性**：高度依赖 LLM 去读写和维护复杂的 Markdown 表格状态，极易因格式错乱导致状态静默丢失。

## 2. 最高优先级：基于 Tauri + Rust 的极致轻量化桌面端架构重构 (Tauri + Rust Architecture)

为了将 Inkos 打造成可商业化发行的独立桌面端软件，同时将安装包体积严格控制在 **30MB 以内**（解决 Node.js + Python RAG 动辄 500MB+ 的体积灾难），系统底层架构必须进行重写。本重构方案为当前开发的第一优先级：

### 2.1 前端：保留 React，引入 Tauri 引擎
*   **抛弃 Electron**：放弃自带 Chromium 内核的 Electron。改用 Tauri 框架，直接调用操作系统原生的 Webview（Windows 的 Edge WebView2 / Mac 的 WebKit），前端体积开销骤降至几 MB。
*   **沉浸式 GUI 渲染**：复用第 11 节设计的“三栏式响应式布局”，Tauri 负责无边框窗口、系统托盘、本地文件系统对话框等原生 OS 能力的交互。

### 2.2 后端：Rust 接管核心管线 (Pipeline & State)
*   **Node.js 逻辑迁移**：将原先基于 Node.js/TypeScript 编写的 `PipelineRunner`、`StateManager` 和多专家 `Auditor` 全部用 Rust 重写。
*   **零运行时依赖**：Rust 编译出的二进制文件极其小巧且运行速度极快。用户电脑无需安装 Node.js 即可直接运行整个 AI 管线。
*   **并发审查优化**：利用 Rust 强大的多线程能力（如 `tokio`），让“逻辑专家”、“人设专家”、“节奏专家”三个 Agent 真正的并行异步会诊，大幅缩短生成和审查的等待时间。

### 2.3 记忆引擎：轻量级本地 RAG (SQLite + 本地向量)
*   **抛弃 Python 与 ChromaDB**：原先的 Python 进程和臃肿的 ChromaDB 向量库是体积失控的最大元凶，必须彻底移除。
*   **SQLite 向量化**：引入 `sqlite-vss` 或 `sqlite-vec` 扩展，将向量检索能力直接内嵌到极度轻量的单文件数据库 SQLite 中。同时接管前文提到的“时序状态机（图结构/JSON Schema）”。
*   **轻量级 Embedding 方案**：
    *   *方案A（纯离线）*：在 Rust 中集成轻量级 ONNX Runtime（如 `candle`），在本地运行极小参数量（~20MB）的 Embedding 模型（如 `bge-micro`），实现纯本地的词向量转换。
    *   *方案B（云端降级）*：如果对包体积有极致的苛求，则调用云端 API（如 OpenAI / 阿里千问 的 Embedding API）进行词向量计算，仅在本地 SQLite 做近似度搜索。

## 3. 核心优化方案与架构设计

为了根治上述问题，我们将对 Inkos 进行深度改造，核心思想是：**“用确定性的代码逻辑接管状态与路由，让大模型只专注文本生成与单点推理。”**

### 3.1 状态管理：引入“时序状态机”与防投毒机制 (Temporal State Engine)
**目标：彻底消灭“死者苏生”和物品归属幻觉。**

*   **结构化真相源 (Structured Truth)**：
    废除直接用 Markdown 表格作为唯一状态源的做法。引入本地轻量级图数据库（或严格的 SQLite + JSON Schema）来管理 `current_state`（角色存活、物品归属、物理位置）。
*   **时序掩码 (Temporal Masking)**：
    为所有实体状态加上生命周期（`Valid-From-Chapter`, `Valid-To-Chapter`）。
    *   *机制*：当角色在第 50 章死亡，其状态的 `Valid-To` 设为 50。
    *   *拦截*：当第 60 章的 Prompt 试图召回该角色时，RAG 层直接拦截或硬性打上 `[已死亡]` 标签，从物理层面切断过期状态对 LLM 的污染。
*   **状态突变审查 (State Mutator Agent)**：
    在章节正文生成后，增加一个微型 Agent，强制要求它输出本章的“状态变更 JSON”（谁死了、谁得到了什么）。代码层校验该变更是否合法（例如：不能丢弃自己没有的物品），校验通过后才允许更新全局数据库。

### 3.2 检索增强：基于意图的动态上下文组装 (Intent-Driven RAG)
**目标：解决 OOC 和上下文注意力衰减。**

*   **出场角色精准切片 (Dynamic Cast Filtering)**：
    在 `PipelineRunner.planChapter` 阶段，强制 Planner 输出本章的**预期出场角色名单**。
    在组装 Writer 的上下文时，**只向 RAG 查询这些特定角色**的《人物口吻与行为圣经》（核心信念、底线、口头禅），将其作为最高优先级的硬约束（Hard Guardrails）注入 Prompt。
*   **反向视角过滤 (Anti-Knowledge / POV Filter)**：
    为了防止 AI 开启“上帝视角”（主角知道了他不该知道的秘密），在检索时引入 `<forbidden_knowledge>` 标签。明确告诉大模型当前视角角色**“不知道什么”**，强力压制剧透幻觉。

### 3.3 审计管线：多专家会诊与主动债务路由 (Multi-Expert & Active Routing)
**目标：解决伏笔遗忘和 Auditor 橡皮图章化。**

*   **微服务化审计 (Micro-Auditing)**：
    将庞大的 `ContinuityAuditor` 拆分为三个并行的轻量级专家：
    1.  **逻辑与战力专家**：对照物理位置和战力设定，专查设定崩坏。
    2.  **人设与视角专家**：对照本章出场人物的《口吻圣经》，逐句排查 OOC 和万能 AI 腔。
    3.  **节奏与爽点专家**：专查本章是否完成了“压抑->释放”的情绪节拍。
    三个专家并行工作，最后由代码层汇总生成精准到段落行号的审查报告。
*   **手术刀式局部精修 (Local Inpainting)**：
    Reviser Agent 不再重写整章，而是根据审查报告提供的段落索引，仅对有问题的段落进行“就地替换”，最大程度保护已有的优秀文本。
*   **主动债务路由 (Active Debt Router)**：
    在生成大纲前，代码层主动扫描《故事链管理台账》和 `pending_hooks`，计算每个伏笔的“滞后值（Staleness）”。
    *   *机制*：如果某个“主线目标链”或“伏笔”超过 3 章未推进，系统不依赖 LLM 的自觉，而是**用代码强行将其注入** Planner 的 `mustAdvance` 数组，作为本章必须解决的“硬任务”。写完后用代码硬性校验是否包含了该伏笔的回收。

## 4. 实施路径 (Implementation Phasing)

由于确立了“极致轻量化桌面端”为最高优先级，我们将对落地路线进行调整，分三个核心阶段推进：

*   **Phase 1: 桌面端脚手架与本地数据库 (Tauri + Rust + SQLite)**
    *   搭建 Tauri + React 三栏式沉浸 GUI 骨架。
    *   在 Rust 后端集成 `sqlite-vss`/`sqlite-vec`，替代庞大的 Python ChromaDB。
    *   实现本地纯 Rust 的基础状态机读写。
*   **Phase 2: 核心管线 Rust 化与多专家并发 (Core Pipeline in Rust)**
    *   用 Rust 重写 `PipelineRunner` 与 `StateManager`。
    *   利用 Rust 的 `tokio` 异步框架，实现 `LogicAuditor`、`CharacterAuditor` 的多线程并发会诊。
*   **Phase 3: 意图驱动检索与主动路由 (Intent RAG & Active Router)**
    *   在 Rust 中实现基于“时序掩码”和“出场角色动态过滤”的本地 RAG。
    *   引入伏笔滞后值计算引擎，实现主动债务路由。

## 5. 文笔与剧情的高阶约束设计 (Advanced Prose & Plot Enforcement)

为了彻底消除 AI 生成的“平铺直叙”和“机器味”，除了底层的状态管理，我们必须在 Planner 和 Reviser 层引入更细粒度的、可代码量化的文学规则。

### 5.1 文笔层：反 AI 模式与具象化约束
*   **"Show, Don't Tell" 的硬性拦截器 (Anti-Tell Dict)**：
    *   **机制**：在 Reviser 阶段引入“AI 标记词黑名单”（如：感到、觉得、似乎、不仅、时间飞逝）。
    *   **拦截**：当检测到词频超标，Reviser 触发 `Spot-fix`（定点修复），强制要求 AI 将抽象情绪转化为物理交互（如把“他感到愤怒”重写为“他捏碎了茶杯”）。
*   **感官锚点法则 (Sensory Anchor Injection)**：
    *   **机制**：Planner 生成大纲时，根据场景地点强制附加 `sensory_directive`（感官指令）。
    *   **约束**：例如规定“本章发生在地牢，必须包含滴水声、霉味或火把的光影描写”。若正文未扫描到对应感官词，直接打回。
*   **段落长度漂移检测 (Paragraph Length Drift)**：
    *   **机制**：通过代码计算正文的段落长度方差，打破 AI “每段固定 3-4 行”的机械排版。
    *   **约束**：强制要求单章内必须出现“单句成段”（用于突发转折或强调）以增加行文的呼吸感。

### 5.2 剧情层：动态节奏与债务驱动
*   **情绪弧线强制反转 (Emotional Arc Enforcement)**：
    *   **机制**：在状态管理器中追踪当前卷的“情绪偏置（Valence）”。
    *   **约束**：若连续两章 Valence 为正（爽点/胜利），下一章 Planner 将强制注入高优先级指令：“禁止平铺直叙，本章必须引入新压抑（挫折/意外）”，防止情绪一顺到底导致读者疲劳。
*   **“代价必须落地”的交易机制 (Consequence Ledger)**：
    *   **机制**：当主角获得重大升级（如越级反杀、获得神器）时，状态机强制 AI 记录一个“等价交换的代价”（如寿命缩短、暴露位置）。
    *   **约束**：该代价转化为一个高优先级的“未爆弹”，在随后的 3-5 章内强制要求 Planner 将其引爆。

## 6. 商业化写作平台全景功能补全 (Commercial-Grade Platform Features)

目前的 Inkos 设计已经具备了“工业级底层引擎”的硬核实力（防幻觉、强逻辑）。要将其打造为真正超越 Scrivener、NovelCrafter 或橙瓜码字的“下一代网文平台”，必须在交互、商业化和人机协同层面补全以下高级特性：

### 6.1 可视化宏观掌控 (Visual Story Management)
*   **多维时间线与故事看板 (Interactive Timeline & Corkboard)**：
    *   将底层的《故事链管理台账》可视化。作者可以通过拖拽卡片排布章节，并在时间轴上直观看到“伏笔滞后值”（如某条红色的线代表该伏笔已 5 章未推进，处于“红温”报警状态）。
*   **动态人物关系与势力图谱 (Dynamic Relationship Web)**：
    *   基于底层图数据库，生成可视化的关系网。拖动时间轴（例如拨到第 50 章），图谱会自动演变（展示谁和谁结仇，哪个门派被灭），帮助作者在宏观层面防止战力与势力崩坏。

### 6.2 创作者生产力与人机共创 (Author Productivity & Co-Pilot)
*   **平行宇宙剧情分支管理 (Git-like Story Branching)**：
    *   在关键剧情节点（如主角是否要杀反派），允许作者创建“What-if 分支”。AI 可以分别推演两种选择对后续 10 章的蝴蝶效应影响，供作者对比后合并最优分支。
*   **伴随式沉浸编辑器 (Immersive Co-Pilot Editor)**：
    *   除了“一键全自动生成”，提供细粒度的编辑器交互：作者可以划选任意一段干瘪的文字，悬浮菜单提供“增强感官细节”、“让语气更阴暗”、“加快动作节奏”等一键原地扩写/重写功能。
*   **一键统一名词与黑屋模式 (Utility Kit)**：
    *   包含网文刚需的“无干扰小黑屋模式”、多端云同步，以及基于 RAG 的全局专有名词一键替换/统一工具。

### 6.3 商业化数据分析与受众对齐 (Market Analytics & Feedback)
*   **AI 读者弹幕模拟系统 (Simulated Reader Reactions)**：
    *   在发布前，利用大模型模拟典型网文读者（如“无脑爽文受众”或“逻辑考究党”）阅读该章。在文档侧边栏生成模拟的“本章说/弹幕”，提前向作者预警潜在的“毒点”（如：这段太憋屈了、配角疯狂抢戏）。
*   **商业化指标与断章点评估 (Cliffhanger & Pacing Analytics)**：
    *   扫描章节的情绪张力和信息密度，给出“商业维度的评分”。利用 AI 算法自动寻找该章的“最高悬念点”，并建议作者在此处“断章”以最大化追读率。
*   **平台合规与敏感词护盾 (Compliance Shield)**：
    *   内置针对国内主流网文平台（起点、番茄等）的涉黄、涉政、暴力等违禁词库扫描，提供一键替换建议，防止封书。

## 7. 爆款包装与市场洞察 (Market & Packaging Tools)
除了内容创作，网文的商业成功高度依赖包装与市场契合度。
*   **爆款书名与简介生成器 (Title & Blurb Optimizer)**：针对起点、番茄等平台的算法推荐逻辑，结合当前书籍大纲，自动生成多套吸睛的书名、简介和标签（Tags），并给出“吸量预期”评分。
*   **网文风向与热梗雷达 (Trend & Tropes Radar)**：系统定期分析全网热门榜单，在作者规划大纲时提供热点建议（例如：“当前榜单流行‘苟道’与‘反套路’，建议在第三卷引入相关元素”）。

## 8. 深度世界观与数值引擎 (Worldbuilding & Mechanics Engine)
长篇玄幻/科幻极易在后期出现逻辑和数值崩盘。
*   **战力与经济体系推演器 (Power & Economy Balancer)**：建立数值模型（如等级经验、货币购买力）。当主角跨级战斗或获得巨额财富时，系统通过图表展示战力曲线与通胀率，预警“战力崩坏”或“金币贬值”。
*   **可视化设定集与地图生成 (Visual Asset Generator)**：联动文生图（Text-to-Image）模型，自动根据描述生成角色立绘、宗门地图、神器草图。不仅帮助作者具象化想象，还可直接导出作为读者粉丝群的运营物料。

## 9. 创作心流与多端协同 (Workflow & Collaboration)
为专业全职作者和工作室提供极致的效率工具。
*   **RAG 增强型语音码字 (Context-Aware Dictation)**：很多作者习惯语音输入。结合书籍专属的 RAG 名词库，系统能精准识别出拗口的“功法名、门派名、角色名”，避免普通语音软件的大量错别字。
*   **工作室风格统一与洗稿检测 (Ghostwriter Style Alignment)**：针对工作室多开或代写场景，AI 可自动扫描代写者的章节，检测并修正其文风，确保与主笔（或前文）的用词习惯高度一致。同时内置原创度查重，防止 AI 或代写引入抄袭片段。

## 10. 资产沉淀与多形态导出 (Export & IP Incubation)
*   **多平台自适应排版 (Multi-Platform Exporter)**：不同小说平台对单章字数（如 2000 字 vs 4000 字）、空行排版有不同要求。提供一键重排功能，甚至直接导出为 EPUB/PDF 实体书格式。
*   **互动小说与短剧脚本转换 (IP Adaptation Exporter)**：一键将小说的高潮章节转化为“短剧分镜头脚本”，或将带有 What-if 分支的剧情导出为 AVG 互动游戏脚本（如 Twine 格式），赋能 IP 多维变现。

## 11. 沉浸式创作 GUI 重构设计 (Immersive Studio UI/UX Design)
当前的 Studio 页面偏向于“程序员视角的后台管理系统”，不符合网文作者的“心流创作”习惯。为此，我们将对 GUI 进行全面重构，全面对标并超越 Scrivener 与 NovelCrafter，采用**“三栏式响应式布局 + 悬浮 AI 伴写”**的设计范式。

### 11.1 核心布局：经典三栏架构 (Three-Column Layout)
界面分为左、中、右三大核心区域，支持自由折叠与拖拽调整宽度：

*   **左侧栏：全局导航与资产库 (Navigator & Assets)**
    *   **大纲树 (Binder)**：多层级的卷、章、节树状目录，支持拖拽排序。章节点缀状态小圆点（如：🔴 待重写、🟡 审计中、🟢 已定稿）。
    *   **世界观百科 (World Bible)**：分类展示角色、地点、势力、物品。支持悬浮预览（Hover Preview），无需跳转页面即可查看角色设定。
    *   **故事看板 (Plot Board)**：切换至软木板视图，以卡片形式全局鸟瞰大纲与伏笔。

*   **中栏：沉浸式编辑器 (Immersive Editor)**
    *   **禅模式 (Zen Mode)**：一键隐藏左右侧栏，全屏专注码字。打字时当前行高亮，背景变暗。
    *   **隐形 AI 交互 (Invisible Co-pilot)**：摒弃右侧死板的 Chat 框，改为**划词悬浮菜单**。选中段落后弹出快捷指令：“加强感官描写”、“缩写”、“语气更狠毒”、“修复病句”。
    *   **内联诊断波浪线 (Inline Diagnostics)**：像代码 IDE 一样，当 AI 审查出 OOC 或逻辑 Bug 时，在文本下方显示波浪线。Hover 即可查看专家 Agent 的修改建议并一键 Apply。
    *   **排版与视觉**：提供专为长文优化的排版（如段首自动缩进、护眼纸张色、行高自定义、多款正版授权中文字体切换）。

*   **右侧栏：上下文与智能面板 (Context & Inspector)**
    *   **当前章控台 (Chapter Inspector)**：展示本章的大纲卡片、核心目标、情绪偏置（Valence）。
    *   **动态上下文 (Dynamic Context)**：**杀手级功能**。随着中栏光标的移动或打字，右侧栏会自动利用 RAG 提取当前段落提到的“角色卡片”和“前置伏笔”，作为备忘录实时展示，作者无需手动搜索。
    *   **债务与伏笔追踪 (Hooks Tracker)**：显示当前强制需要推进的“待回收伏笔（Pending Hooks）”，并在写到相关内容时自动打钩。
    *   **多专家审查报告 (Audit Panel)**：展示本章的整体健康度评分（逻辑、人设、节奏），并列出详细的修改建议列表。

### 11.2 交互范式升级
*   **指令面板 (Command Palette)**：引入类似 VSCode 的 `Ctrl+K` / `Cmd+K` 全局搜索与指令输入框。作者可以随时呼出输入：“新建角色 楚风”、“查找楚风第一次使用拔剑术的章节”，全程键盘操作，无需动鼠标。
*   **版本历史与时光机 (Time Machine)**：由于 AI 生成的内容可能不尽如人意，提供类似 Git 的节点对比视图。作者可以并排比对“AI 修改前”和“AI 修改后”的文本，支持精确到句子的挑选合并。
*   **状态栏 (Status Bar)**：底部常驻状态栏，实时显示今日码字字数、历史平均时速、本章目标字数进度条，以及当前的系统状态（如：🤖 RAG 引擎就绪、📡 模型连接正常）。
