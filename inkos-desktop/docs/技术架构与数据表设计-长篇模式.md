# 技术架构与数据表设计：长篇模式

## 1. 文档信息

- 文档名称：`NovelOS Longform 技术架构与数据表设计`
- 文档类型：技术设计文档
- 面向对象：后端工程师、算法工程师、数据库工程师、前端架构师
- 目标：为 200 万字级长篇模式提供可实现的系统架构、数据模型和接口契约

## 2. 设计目标

- 在本地优先的写作环境中稳定支撑超长篇项目
- 保证“事实基准”和“文本生成”解耦
- 让每一章在生成、校验、归档、回滚上都有确定流程
- 支持卷级、事件链级和章节级多粒度状态恢复
- 支持后续扩展到服务端同步和多用户协作

## 3. 设计原则

- `事实入库优先于文本留存`
- `状态快照优先于长上下文拼接`
- `规则编译优先于经验判断`
- `版本化优先于覆盖式修改`
- `最小影响范围优先于全书重算`

## 4. 总体技术架构

建议采用以下逻辑分层：

- `Client Layer`
  - Tauri/Web 前端
  - 编辑器、项目看板、正典中心、修史审批台
- `Application Layer`
  - 工作流编排器
  - 项目服务、章节服务、修史服务、审计服务
- `Domain Layer`
  - 正典引擎
  - 快照引擎
  - 连续性编译器
  - 召回引擎
  - 健康度分析器
- `Data Layer`
  - SQLite 本地数据库
  - 向量索引或全文索引
  - 文件型归档存储（章节正文、diff、导出）
- `LLM Layer`
  - 规划模型
  - 正文模型
  - 审计模型
  - 重写模型

## 5. 核心运行链路

### 5.1 章节生成链路

- 读取当前卷、篇章、事件链状态
- 召回本章相关正典、状态、伏笔、未解因果
- 生成章节任务卡
- 调用正文模型生成初稿
- 调用连续性编译器执行规则检查
- 调用评审/重写模型定向修复
- 定稿后写入章节表与账本表
- 生成新快照

### 5.2 修史链路

- 用户创建修史申请
- 系统解析变更对象和规则类型
- 影响分析器计算受影响卷/章/对象
- 生成修复方案
- 用户审批执行
- 对受影响区间重新审计并更新快照

## 6. 服务模块设计

### 6.1 Project Service

职责：

- 创建长篇项目
- 管理项目设置、目标总字数、卷数量、模式开关

### 6.2 Canon Service

职责：

- 管理正典对象
- 区分硬规则、软规则、开放规则
- 处理版本冻结和解冻

### 6.3 Outline Service

职责：

- 管理作品级、卷级、篇章级、事件链级和章节级规划
- 提供章节任务卡输入

### 6.4 Chapter Service

职责：

- 管理章节草稿、定稿、版本、比较、回滚
- 维护章节状态机

### 6.5 Snapshot Service

职责：

- 生成章节快照、事件链快照、卷级快照
- 为召回服务提供轻量事实基准

### 6.6 Continuity Compiler Service

职责：

- 读取规则与事实
- 计算冲突和风险
- 输出结构化问题列表

### 6.7 Retcon Service

职责：

- 接收修史申请
- 执行影响分析
- 协调回写和回归测试

### 6.8 Recall Service

职责：

- 根据当前写作任务精准召回相关信息
- 控制单次 prompt 输入规模

## 7. 数据库选型建议

### 7.1 本地优先方案

- 数据库：SQLite
- 原因：
  - 已适配 Tauri 本地应用
  - 写作项目以单用户本地编辑为主
  - 支持事务、索引、FTS、版本化快照

### 7.2 后续扩展方案

- 本地 SQLite + 远端 Postgres 同步
- 适用于工作室协作、版本云备份和跨设备访问

## 9. FTS 与检索建议

### 9.1 全文检索

建议建立 FTS 表：

- `chapters_fts`
- `canon_rules_fts`
- `event_nodes_fts`

支持查询：

- 某个秘密第一次出现在哪章
- 某个角色何时提到某个地点
- 某类规则在何时被引用

### 9.2 召回策略

召回服务不直接全量拼接上下文，而是按以下顺序构造输入：

- 当前卷目标
- 当前事件链摘要
- 当前章节任务卡
- 最近快照
- 必需正典条目
- 必需人物状态
- 相关伏笔和未解因果

## 10. 连续性编译器设计

### 10.1 输入

- 当前章节草稿
- 当前章节任务卡
- 最近快照
- 相关正典规则
- 相关状态表记录

### 10.2 输出结构

```json
{
  "status": "pass|warning|fail",
  "score": 0.0,
  "issues": [
    {
      "type": "canon_conflict|timeline_conflict|character_drift|power_drift|visibility_error|foreshadow_issue",
      "severity": "low|medium|high|critical",
      "message": "string",
      "evidence_refs": ["string"],
      "suggested_action": "rewrite|update_state|retcon_request"
    }
  ]
}
```

### 10.3 编译子模块

- `CanonChecker`
- `CharacterChecker`
- `TimelineChecker`
- `PowerChecker`
- `VisibilityChecker`
- `ForeshadowChecker`

## 12. 后台任务设计

### 12.1 同步任务

- 章节定稿后自动写账本
- 章节定稿后自动生成快照
- 修史执行后自动重新编译受影响章节

### 12.2 异步任务

- 长篇健康度巡检
- 伏笔滞后扫描
- 人物漂移扫描
- 修史影响分析

## 13. 性能与规模设计

### 13.1 目标规模

- 章节规模：5000 章
- 总字数规模：200 万到 800 万字
- 角色规模：100 到 1000
- 伏笔规模：500 到 5000

### 13.2 性能原则

- 所有章节列表必须虚拟滚动或分页
- 卷级和事件链查询必须走索引
- 快照应优先用于召回，避免重复扫描全书
- 编译器优先做增量检查，不做全书全量扫描

## 14. 容错与回滚

### 14.1 事务边界

以下操作必须事务化：

- 章节定稿 + 账本写入 + 快照生成
- 正典更新 + 版本记录
- 修史审批 + 影响记录 + 状态更新

### 14.2 回滚策略

- 章节回滚：按 `chapter_versions`
- 正典回滚：按 `canon_rule_versions`
- 修史回滚：按 `retcon_requests` 和受影响对象版本

## 15. 安全与权限

### 15.1 本地模式

- 单用户写作，无复杂权限模型

### 15.2 协作扩展

- 角色建议：作者、编辑、总编、管理员
- 硬规则冻结和修史审批需要高级权限

## 16. 监控与埋点

建议埋点：

- 每章初稿生成耗时
- 编译失败原因分布
- 一次通过率
- 修史申请频率
- 快照召回命中率
- 人工采纳率

## 18. 总结

长篇模式的技术重点不是把模型接进去，而是先构建“事实数据库、状态快照、连续性编译器、修史治理链路”四个底座。只要这四个底座稳定，正文模型就能在一个可控的系统里工作，长篇反幻觉才真正具备工程可行性。
