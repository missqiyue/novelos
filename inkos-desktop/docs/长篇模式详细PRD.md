# 长篇模式详细 PRD

## 1. 文档信息

- 产品名称：`NovelOS Longform`
- 文档类型：产品需求文档（PRD）
- 适用范围：100 万字以上长篇连载，重点支持 200 万字以上超长篇网文
- 目标读者：产品经理、项目负责人、内容总编、算法负责人、研发负责人

## 2. 背景与问题

现有 AI 写作工具在短篇、中篇或前几十章阶段通常还能维持可读性，但一旦进入 100 万字以上的长篇连载，就会快速暴露以下问题：

- 设定随生成过程漂移，早期硬规则被后文推翻
- 角色状态失真，人物说话、行事、立场和前文不一致
- 时间线失控，跨度、位置、事件先后顺序反复冲突
- 伏笔大量埋下但没有回收，或者提前回收、错误回收
- 战力体系和资源消耗不再守恒，导致爽点廉价化
- 局部章节看似通顺，但全书级因果已经断裂

因此，长篇模式的核心问题不是“如何继续生成更多文字”，而是“如何在超长文本规模下持续保持世界、人物、因果与节奏的一致性”。

## 3. 产品目标

### 3.1 核心目标

- 支持用户用结构化方式管理 200 万字级长篇项目
- 让 AI 在受控约束下持续生成章节，而不是自由扩写
- 显著降低设定冲突、人物漂移、时间线冲突和伏笔遗失
- 让每一章在定稿前都经过“编译式审计”
- 形成可追踪、可回滚、可修史、可版本化的长篇生产流程

### 3.2 用户可感知目标

- 用户可以随时知道“这本书现在的正典是什么”
- 用户可以在写新章前看到角色、关系、战力、时间和伏笔的当前状态
- 用户可以在大纲变更时清楚知道会影响哪些卷和章节
- 用户可以在 500 章之后仍然稳定调取前 50 章的重要事实

### 3.3 非目标

- 不追求全自动零人工产出完整爆款长篇
- 不在 V1 解决跨平台发布和商业结算问题
- 不在 V1 支持多作者实时协同编辑
- 不在 V1 构建复杂图数据库可视化编辑器

## 4. 用户画像

### 4.1 核心用户

- 商业网文作者：有稳定更新诉求，希望 AI 提高产能但不希望人设崩坏
- 编辑工作室：需要同时管理多个长篇项目，并对质量和一致性负责
- 平台定制团队：需要批量生成题材稳定、结构清晰的长篇项目

### 4.2 典型场景

- 场景 A：用户准备写第 438 章，需要先确认主角当前境界、最近关系变化和未回收伏笔
- 场景 B：用户打算修改第 2 卷核心设定，系统需要评估对第 3 至第 7 卷的影响
- 场景 C：用户发现后期节奏乏力，希望系统从卷级和事件链级别给出修复建议
- 场景 D：用户需要在 2000 章规模下快速搜索“谁知道某秘密、何时知道、后续是否触发”

## 5. 核心定义

### 5.1 长篇幻觉定义

长篇模式中的“幻觉”不是泛指语言错误，而是以下七类结构性错误：

- 设定幻觉：世界规则、体系边界、组织结构与前文冲突
- 人物幻觉：角色行为、台词、价值观与既有设定冲突
- 时间幻觉：事件顺序、时长跨度、年龄与地理移动冲突
- 因果幻觉：事件结果无起因，或者铺垫已做但结果丢失
- 战力幻觉：升级、资源、技能限制突然失效
- 情绪幻觉：关系与情绪变化没有足够触发事件
- 记忆幻觉：已公开信息、已埋伏笔或旧人物被系统“遗忘”

### 5.2 正典（Canon）定义

- 正典是当前项目中被系统承认为“事实基准”的内容集合
- 正典包含：硬设定、角色基础档案、卷级确认结论、已定稿章节事实、状态快照
- 未进入正典的内容一律视为提案，不得直接作为后续章节的事实前提

## 6. 产品设计原则

- `规划优先`：先冻结终局承诺和阶段目标，再进行章节生产
- `账本优先`：所有关键事实必须入库，而不是只存在正文
- `状态优先`：系统更重视“当前状态”而非静态简介
- `审计优先`：章节生成后先检查再归档，不允许直接连写
- `最小修改优先`：出现冲突时优先修当前链路，尽量不大规模改旧文
- `人机分权`：AI 可提案，正典冻结与修史审批由用户或编辑确认

## 8. 信息架构

长篇模式新增 6 个一级模块：

- `正典中心`
- `长程剧情树`
- `故事账本`
- `快照中心`
- `连续性编译器`
- `修史审批台`

现有模块继续保留并升级：

- 小说圣经
- 节奏规划器
- 章节工作台
- 评审中心
- 发布包装台

## 9. 关键功能设计

### 9.1 正典中心

目标：统一管理所有硬设定和冻结规则。

功能：

- 维护世界底层规则、力量天花板、禁忌规则、系统边界
- 管理角色不可违背边界、核心身份、核心关系
- 标记字段类型：`硬规则`、`软规则`、`开放规则`
- 提供版本冻结、变更日志和修史申请入口

验收标准：

- 任意硬设定都能追溯创建来源和最后修改时间
- 正文生成前可以自动召回与本章相关的硬规则

### 9.2 长程剧情树

目标：把 200 万字拆成可控层级。

层级结构：

- 作品级：总卖点、主线、终局、主题母线
- 卷级：每卷目标、核心反派、阶段成长、卷末爆点
- 篇章级：20 到 50 章组成一个篇章
- 事件链级：3 到 10 章完成一个明确事件闭环
- 章节级：单章目标、冲突、信息、高潮、尾钩

核心规则：

- 章节不可脱离事件链单独生成
- 事件链必须隶属于某卷目标

### 9.3 故事账本

目标：将长篇写作中的关键事实结构化保存。

账本类型：

- 世界规则账
- 人物状态账
- 关系状态账
- 时间线账
- 事件账
- 伏笔账
- 信息可见性账
- 资源与代价账

核心能力：

- 事实追溯
- 状态检索
- 冲突定位
- 条件筛选

### 9.4 快照引擎

目标：为超长篇提供稳定的“中间记忆”。

快照粒度：

- 章节快照：每章定稿后生成
- 事件链快照：每个事件链结束后生成
- 卷级快照：每卷结束后生成

快照包含：

- 活跃主线/支线
- 主要角色状态
- 关系变化
- 已公开信息
- 未回收伏笔
- 节奏方向

### 9.5 连续性编译器

目标：像代码编译器一样拦截结构性错误。

检查项：

- 设定冲突
- 人物口吻偏移
- 时间线冲突
- 战力跳变
- 信息越权
- 伏笔错误回收

输出结果：

- 通过
- 警告
- 失败

失败处理：

- 阻止章节归档
- 生成问题列表
- 推荐回退位置与修复策略

### 9.6 修史审批台

目标：控制大纲或硬设定变更的影响面。

功能：

- 发起修史申请
- 评估受影响卷/章/角色/伏笔
- 给出三种修复方案：后续补偿、局部回写、卷级重构
- 记录审批结论与执行版本

### 9.7 长篇健康度看板

目标：让用户看到项目是否正在“慢性失控”。

核心指标：

- 世界稳定度
- 人物稳定度
- 时间线稳定度
- 伏笔兑现率
- 设定膨胀率
- 修史频率
- 章节编译通过率
- 高风险漂移率

## 10. 页面设计

### 10.1 长篇项目首页

- 顶部展示项目状态、卷进度、健康度总览
- 中部展示当前卷、当前事件链、待处理修史与高风险章节
- 右侧展示最近快照摘要

### 10.2 正典中心页

- 左侧设定树
- 中间设定详情
- 右侧影响范围与历史版本

### 10.3 长程剧情树页

- 按作品/卷/篇章/事件链/章节五级展开
- 支持折叠、拖拽调整、批量重排

### 10.4 章节工作台

- 顶部显示本章任务卡
- 侧栏显示相关正典条目、角色状态、伏笔提示
- 下方显示编译结果与待修复问题

### 10.5 修史审批台

- 展示申请原因、影响范围、风险等级、推荐方案
- 支持人工审批和回退比较

## 11. 核心流程

### 11.1 新建长篇项目

- 输入作品定位、题材、目标总字数、预计卷数
- 生成作品级总纲与终局承诺
- 生成卷纲草案
- 用户确认后冻结作品级正典

### 11.2 章节生产流程

- 读取当前卷目标和事件链目标
- 读取本章相关正典与状态快照
- 生成章节任务卡
- 生成章节初稿
- 跑连续性编译器
- 若通过则进入评审和修订
- 定稿后写入账本并生成快照

### 11.3 修史流程

- 用户提出变更
- 系统识别是否涉及硬规则
- 生成影响分析报告
- 用户选择修复策略
- 执行后对受影响区间做回归检查

## 12. 规则系统

### 12.1 强制规则

- 正文不得直接新增硬设定
- 主角核心能力变化必须在账本中存在前置铺垫
- 每章必须绑定至少一个事件链
- 每个重要关系变化必须绑定触发事件
- 每卷结束必须生成卷级快照

### 12.2 风险规则

- 长时间未出现的重要角色触发状态刷新提醒
- 某类设定新增过快时触发“设定膨胀”警报
- 某伏笔滞后过久时触发“延期回收”提醒

## 13. 指标体系

### 13.1 产品指标

- 长篇项目完备率
- 快照生成成功率
- 修史申请完成率
- 用户周活跃编辑时长

### 13.2 内容指标

- 设定冲突数 / 万字
- 人物偏移数 / 百章
- 时间线冲突数 / 卷
- 伏笔回收及时率
- 编译一次通过率

### 13.3 业务指标

- AI 初稿采纳率
- 人工修改时长
- 卷级返工率
- 中后期章节质量衰减率

## 14. 里程碑规划

### Phase 1：长篇底座（4 周）

- 长篇项目类型
- 正典中心基础版
- 卷纲和事件链管理
- 章节任务卡

### Phase 2：状态与快照（4 周）

- 故事账本
- 章节快照与卷级快照
- 长篇健康度基础指标

### Phase 3：编译器与修史（4 周）

- 连续性编译器
- 修史审批流程
- 回归检查

### Phase 4：优化与灰度（3 周）

- 风险阈值调优
- 指标回流
- 样板项目试跑

## 15. 风险与应对

- 风险：结构过重导致用户学习成本高
  - 应对：提供 `标准模式` 与 `专家模式`
- 风险：规则过严抑制创造性
  - 应对：区分硬规则和软规则，允许爆点章更高自由度
- 风险：修史链路太复杂
  - 应对：优先推荐“后续补偿”式最小修复
- 风险：数据项过多造成维护负担
  - 应对：优先记录高价值事实，低价值细节只做摘要

## 16. 验收标准

- 能完成一个 8 卷、200 万字级项目的结构初始化
- 能在 1000 章规模下稳定检索正典、状态与伏笔
- 任意章节定稿前都必须经历编译检查
- 能对一次硬设定修改输出明确影响分析
- 卷级快照可用于后续章节生成并有效减少一致性冲突

## 17. 总结

长篇模式不是单纯的“加大上下文”，而是把小说生产升级为一套具备正典管理、状态管理、快照管理、编译审计和修史治理能力的内容操作系统。只有这样，AI 才能在 200 万字以上的连载中持续可控，而不是越写越乱。
