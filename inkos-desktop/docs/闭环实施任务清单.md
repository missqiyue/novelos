# 0. 范围与目标

本文把 [PlanningContext-闭环设计.md](file:///Volumes/MOVE/AI/work/writing/%E8%87%AA%E5%8A%A8%E5%8C%96/inkos-desktop/docs/PlanningContext-%E9%97%AD%E7%8E%AF%E8%AE%BE%E8%AE%A1.md) 的设计拆成“可执行条目”，按以下顺序推进：

1) 必须先保留 raw draft（防上下文丢失）
2) 再注入 PlanningContext（让蓝图约束写章）
3) 再引入 ComplianceReport / RewriteRequest（可验证 + 可纠偏）
4) 最后补 checkpoint 派生更新与 gate（版本/锁定/定稿门槛）

约定：

- 每条任务都包含：目标、涉及模块、验收点
- 不写具体代码实现，但任务描述足够让实现者直接开干
- 优先级：P0 必做 / P1 应做 / P2 可选

# 1) 先保留 raw draft（P0）

## 1.1 写章生成保留完整三段输出

- 目标：写章后能拿到完整 `PRE_WRITE_CHECK/CHAPTER_TITLE/CHAPTER_CONTENT`，为后续校验与重写提供输入
- 涉及模块：
  - Tauri: `generate_chapter_pipeline` 的返回值策略（目前会只返回正文）
  - 前端：写章编辑器的数据结构（保存 raw draft 与 parsed 片段）
- 建议方案（二选一，推荐 A）：
  - A. 让 `generate_chapter_pipeline` 返回 raw draft，同时新增 `extract_*` 在前端解析并分别保存 title/content/pre_write_check
  - B. 维持返回正文，但新增另一个命令 `generate_chapter_draft_raw` 专门返回 raw draft，并在前端调用它
- 验收点：
  - DB 中能保存 raw draft（至少本地 state；更推荐落库）
  - UI 能展示/复制 PRE_WRITE_CHECK 与标题

## 1.2 数据库存储：增加 draft_raw 字段或独立表

- 目标：raw draft 可追踪、可回放、可用于回归对比
- 涉及模块：
  - book db schema：`chapters` 表扩字段，或新表 `chapter_drafts(chapter_number, draft_raw, created_at, source)`
- 验收点：
  - 同一章可保留最新 draft_raw（MVP），或保留多版本（增强）

## 1.3 前端写章保存流程对齐

- 目标：写章后保存的不仅是正文，还要保存 outline 与 draft_raw（或至少 pre_write_check）
- 涉及模块：
  - `save_chapter_content/save_chapter_outline`
  - App.tsx 的写章按钮逻辑
- 验收点：
  - 点击“生成正文”后，DB 与 UI 都能回读到 draft_raw 与正文一致

# 2) 再注入 PlanningContext（P0→P1）

## 2.1 新增聚合读取命令 get_planning_context（MVP）

- 目标：写章/评审/提案统一读取同一份上下文，避免各自拼接
- 涉及模块：
  - Tauri: 新 command `get_planning_context(chapter_number, version_id?, limits...)`
  - DB 查询：book_meta / one_liner_current / outline_checkpoints / open_hooks / recent_context
- 验收点：
  - 返回 JSON 包含：book/plan/checkpoint?/open_hooks/recent_context/warnings
  - 缺失数据时 warnings 可提示，但命令不崩

## 2.2 写章 prompt 注入 PlanningContext（MVP：四件套）

- 目标：把闭环从“写后反哺”升级为“写中约束”
- 注入内容（MVP）：
  - `plan.one_liner + locked`
  - `checkpoint`（若存在）
  - `open_hooks`
  - `recent_context`
- 涉及模块：
  - `generate_chapter_outline`（生成大纲时也应引用 one_liner/checkpoint）
  - `generate_chapter_pipeline`（生成正文）
- 验收点：
  - prompt 中能看到 PlanningContext 注入块（可通过日志/调试开关验证）
  - 生成结果稳定包含“伏笔闭环”记录（至少 1 条）

## 2.3 Planning / Binder / Board 页面统一引用 active 版本（P1）

- 目标：避免“最新蓝图版本 ≠ 当前写章引用版本”
- 涉及模块：
  - 引入 `active_plan_version_id`（存 book_meta 或 book_settings）
  - Planning 页显示最新版本 vs active 版本
- 验收点：
  - 写章默认引用 active 版本的 checkpoint/stage_plan

# 3) 再引入 ComplianceReport / RewriteRequest（P0→P1）

## 3.1 升级 PRE_WRITE_CHECK 表格契约（P0）

- 目标：让模型输出可抽取、可自动判定
- 关键变化：
  - 列名固定为：`检查项/本章记录/证据/定位/结果`
  - 必需行至少 6 项（规划锚定/Checkpoint 一致性/线程推进/伏笔闭环/命名一致性/风险扫描）
  - 强制证据标记：`hook_id:<id>`、`thread:<THREAD_KEY>`、`one_liner:`、`violated:...`
- 涉及模块：
  - `generate_chapter_pipeline` 的 system prompt
- 验收点：
  - 任意一章输出均能用简单 parser 提取出必需项与证据标记

## 3.2 新增合规校验器命令 compute_compliance_report（P0）

- 目标：自动判定 PASS/WARN/FAIL，并把失败映射到 action
- 输入：
  - planning_context + draft_raw（或 pre_write_check + refs）
- 输出：
  - ComplianceReport（见设计文档第 10 节）
- 验收点：
  - 能检测：缺区块、缺 hook_id、hook_id 非法、thread_refs 缺 thread 证据等
  - 输出 actions 推荐（至少 rewrite_request / call_command）

## 3.3 新增重写入口 rewrite_chapter_with_constraints（P0）

- 目标：FAIL 后能够“带约束重写”并再次校验
- 输入：
  - RewriteRequest（见设计文档第 11 节）
- 输出：
  - RewriteResult（含新的 draft_raw）
- 验收点：
  - 能完成一次 FAIL→重写→PASS 的闭环演示（MVP 先只修 hook_close）

## 3.4 UI 联动：写章后自动校验与状态流转（P1）

- 目标：减少用户手工步骤，让流程自动推进
- 建议策略（MVP）：
  - 写章生成后自动跑 compliance
  - FAIL：提示原因 + 一键“约束重写”
  - PASS：允许进入评审/定稿
- 验收点：
  - 用户不需要读文档也能走通一章闭环

# 4) 最后补 checkpoint 派生更新与 gate（P1→P2）

## 4.1 outline_patch apply 尊重 locked 语义（P1）

- 目标：锁定章节不被 AI 自动覆盖
- 涉及模块：
  - `accept_proposal` 的 outline_patch 分支
  - `save_outline_patches`（是否对 locked 行做保护，取决于产品策略：锁是否只防 AI）
- 验收点：
  - locked=true 时，AI 提案默认不覆盖 one_liner（可降级为 WARN 或 require manual unlock）

## 4.2 apply 后派生动作：checkpoint_recompute（P1）

- 目标：规划变更后事实基准也能更新，避免下游引用旧 checkpoint
- 触发条件（MVP）：
  - outline_patch 影响 one_liner/thread_refs/cast_refs
  - thread_upsert / soul_timeline_upsert 影响事实基准
- 涉及模块：
  - 新 command `recompute_checkpoint(start,end,version_id)`
  - checkpoint_json schema v0（先弱校验）
- 验收点：
  - 应用提案后能生成新的 checkpoint 记录（可在 UI 查看）

## 4.3 定稿 gate：未通过合规/评审禁止 finalized（P1）

- 目标：把“闭环要求”变成产品门槛，避免设定漂移积累
- 规则（MVP）：
  - overall=FAIL → 禁止 finalized
  - overall=WARN → 提示但允许（可配置）
- 涉及模块：
  - UI：Finalize 按钮逻辑
  - DB：保存 last_compliance_report（推荐）
- 验收点：
  - 明确提示“阻塞原因”，并提供“一键修复”入口

## 4.4 收件箱 gate（P2）

- 目标：避免提案长期积压导致世界观/线程漂移
- 规则示例：
  - 若存在 `proposal_type=conflict/review` 或 `world_facts_proposals` 未处理且达到阈值，则提示或阻塞进入下一章
- 验收点：
  - 用户能在合适时机被提醒处理“冲突类提案”

# 5. 最小里程碑建议（供排期/验收）

- M0（可校验）：raw draft 可保存 + PRE_WRITE_CHECK 契约升级
- M1（写中约束）：get_planning_context + 写章 prompt 注入（MVP 四件套）
- M2（可纠偏）：compliance_report + rewrite_request + 一键重写
- M3（强闭环）：apply 尊重 locked + checkpoint_recompute + finalized gate
