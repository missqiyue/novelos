# NovelOS Longform 完整系统设计

## 1. 文档信息

- 产品名称：`NovelOS Longform`
- 文档类型：产品需求 + 技术架构 + Agent规范 三合一完整设计
- 适用范围：50万字至500万字长篇网文，重点支撑200万字以上超长篇
- 核心命题：让AI在超长文本规模下持续生成"像人写的、像爆款写的、不崩不乱"的网文

## 2. 一句话定义

NovelOS Longform 不是"加大上下文的AI写作器"，而是一套**从用户一句话到百万字成书**的完整内容操作系统——它负责题材识别、模板匹配、文风控制、自动续写、多专家评审、读者反馈闭环、去AI痕迹，多书并行管理互不干扰，以及在200万字规模下维持人物、装备、道具、剧情绝对不乱。

## 第一部分：产品定义

## 3. 核心问题

现有AI写作工具的五个致命缺陷：

1. **AI味重**：生成的文字工整、对称、过度解释，读者一秒识别
2. **类型盲**：不懂玄幻升级流和悬疑探案是两套完全不同的节奏系统
3. **越长越乱**：100章后角色变了、时间线崩了、伏笔丢了、战力通胀了
4. **无法闭环**：写完不能审、审完不能改、改完不能追踪影响
5. **无法学习**：不能从爆款中提取可复用的写作手法

## 4. 产品目标

### 4.1 核心目标

| 编号 | 目标 | 可验证标准 |
|------|------|-----------|
| G1 | 用户一句话即可启动一个完整长篇项目 | 系统在30秒内输出题材识别、模板推荐、卷纲草案 |
| G2 | AI初稿的AI味识别率低于5% | 经Voice Filter后，人工盲测AI检出率<5% |
| G3 | 200万字规模下零硬性冲突 | 编译器拦截所有设定/人物/时间线/战力硬冲突 |
| G4 | 章节必须经过审核才能定稿 | 未通过编译+评审的章节无法归档 |
| G5 | 读者评论可驱动修订 | 读者反馈可转化为修订任务并追踪执行 |
| G6 | 爆款手法可提取可复用 | 爆款解析后生成的模式可被新项目引用 |

### 4.2 非目标

- 不追求全自动零人工产出完整长篇
- V1不做跨平台发布和商业结算
- V1不做多人实时协同编辑
- V1不做图数据库可视化编辑器

## 5. 用户画像

| 用户 | 核心诉求 | 使用频率 |
|------|---------|---------|
| 商业网文作者 | 提高产能、保证不崩、去AI味 | 日活 |
| 编辑工作室 | 管理多项目、把控质量、审核流程 | 日活 |
| 平台定制团队 | 批量生成题材稳定的长篇 | 周活 |
| 新手作者 | 一句话开始、模板引导、学习爆款 | 周活 |

## 6. 典型场景

| 场景 | 描述 |
|------|------|
| S1 一键启动 | 用户输入"修仙升级流，主角从废材逆袭，有系统金手指"，系统自动匹配题材模板、生成世界观框架、为主副角命名和匹配SOUL、生成候选书名、输出8卷卷纲 |
| S2 续写 | 写到第438章，系统自动召回主角当前状态、活跃伏笔、近期关系变化，生成任务卡并续写 |
| S3 审核 | 第439章初稿生成后，连续性编译器检查冲突，三位评审Agent从不同角度审核，全部通过才可定稿 |
| S4 读者反馈 | 读者评论"最近20章主角太圣母了"，系统转化为情绪弧线修订任务，建议下一事件链增加立场冲突 |
| S5 去AI | Voice Filter自动替换"毋庸置疑"等AI痕迹词，重写过度解释段落，调整句式节奏 |
| S6 学爆款 | 导入《诡秘之主》前50章，系统提取悬疑铺设节奏模式，应用到当前项目的悬疑支线 |
| S7 修史 | 用户要改第3卷核心设定，系统分析影响4-7卷12章3条伏笔，推荐最小修复方案 |

## 第二部分：一键启动与题材模板系统

## 7. 一键启动链路

### 7.1 总体流程

```
用户简短描述 → 题材识别Agent → 模板匹配 → 世界观预填充 → 卷纲生成 → 正典冻结 → 项目就绪
```

### 7.2 题材识别

用户输入可以是：
- 一句话描述："都市重生，主角回到2010年炒股"
- 关键词组合："玄幻 + 系统 + 升级"
- 对标作品："类似《斗破苍穹》但主角是女"

系统输出：
```json
{
  "primary_genre": "xuanhuan_upgrade",
  "sub_genres": ["system_golden_finger", "underdog_reversal"],
  "confidence": 0.91,
  "template_id": "tpl_xuanhuan_system_v3",
  "recommended_volumes": 8,
  "recommended_chapters_per_volume": 30,
  "estimated_total_words": 2400000,
  "default_min_chapter_words": 2000,
  "default_max_chapter_words": 5000
}
```

### 7.3 题材模板库

每种题材模板包含：

| 模板组件 | 说明 | 示例（玄幻升级流） |
|---------|------|-----------------|
| 世界观框架 | 力量体系、组织结构、地理层级 | 修仙境界九层、宗门三级制、凡界→灵界→仙界 |
| 卷节奏模板 | 每卷的标准节奏结构 | 铺垫(5章)→冲突(10章)→突破(5章)→收获(5章)→余波(5章) |
| 角色原型库 | 该题材常见角色类型 | 废材主角、冷面师兄、温柔师妹、腹黑反派 |
| 爽点节奏参数 | 爽点密度、间隔、强度曲线 | 每5章一个小爽点，每卷末一个大爽点，爽点强度递增15% |
| 禁忌规则 | 该题材不应出现的东西 | 玄幻中不出现科技武器、升级流不出现长时间无成长 |
| 文风默认值 | 该题材的推荐文风 | 短句为主、动作描写密集、对话简洁有力 |
| 默认章节字数范围 | 该题材的每章字数建议 | 玄幻2000-5000，悬疑2500-4000，言情2500-4500 |
| 套路库 | 该题材的经典叙事套路 | 扮猪吃虎、退婚打脸、拍卖会夺宝、秘境历练 |

### 7.5 自动预填充

题材匹配后系统自动预填充：

1. **世界观骨架**：根据模板生成力量体系、地理层级、组织结构草案
2. **卷纲草案**：根据卷节奏模板生成8卷目标、核心冲突、卷末爆点
3. **角色档案草案**：从角色原型库生成5-8个核心角色档案
4. **角色命名**：由 Name Generator Agent 为主/副角生成易记且不雷同的名称（详见7.6）
5. **角色SOUL生成**：由 SOUL Matcher Agent 为每个角色从内置SOUL库匹配并定制SOUL档案（详见7.7）
6. **书名生成**：由 Book Title Generator Agent 根据描述、题材、大纲等生成3-5个候选书名（详见7.8）
7. **正典草案**：从模板禁忌规则和世界观生成硬规则集合
8. **文风配置**：从模板文风默认值生成文风档案
9. **章节字数范围**：从模板默认章节字数范围生成初始值，用户可调整

用户可以全部接受、局部修改、或从空白开始。

### 7.6 角色命名系统

建书时AI自动为主/副角生成名称，核心要求：**易记但不雷同**。

#### 7.6.1 问题定义

| 现状 | 问题 |
|------|------|
| AI默认命名 | "凌天""萧炎""叶辰"——与已有网文高度雷同，读者一看就出戏 |
| 用户自行起名 | 费时费力，且容易起出"张伟""李明"般毫无特色的名字 |
| 随机字典拼凑 | 名字无语义关联，玄幻世界出现"王建国"违和感强 |

#### 7.6.2 命名原则

1. **题材适配**：名字必须符合世界观语境（修仙≠都市≠末世）
2. **辨识度优先**：读一遍能记住，不用反复回看确认"这人是谁"
3. **不与热门书撞名**：避开《斗破》《凡人》《诡秘》等知名作品的主角/核心配角名
4. **语义暗示**：名字隐含角色特质，但不过于直白（"冷无霜"暗示清冷，"铁锤"太直白）
5. **音韵区分**：同一书中主要角色名不撞声母、不撞韵尾，避免"林枫/林峰/林锋"类混淆

**输入**：

#### 7.6.4 命名风格与题材对应

| 题材 | 姓氏风格 | 名字风格 | 示例 |
|------|---------|---------|------|
| 玄幻升级流 | 古姓偏多（沈、裴、慕容） | 单字/双字均可，偏意境 | 沈牧、裴惊鸿 |
| 仙侠修真 | 道家姓（李、张、陈） | 清雅出尘，可带道号 | 陈忘机、青莲道人 |
| 都市重生 | 现代常见姓 | 日常但不平庸 | 陆行舟、江一帆 |
| 系统金手指 | 不限 | 可带点"系统味"的利落感 | 方锐、周策 |
| 悬疑探案 | 不限 | 偏冷硬或知识分子感 | 顾深、温予安 |
| 历史穿越 | 朝代对应姓氏 | 符合时代 | 赵北望、萧归 |
| 甜宠言情 | 偏雅致 | 女名柔美男名清朗 | 苏念晚、顾行知 |
| 虐恋情深 | 可带悲剧暗示 | 名字本身有宿命感 | 江离、沈无归 |
| 科幻末世 | 可中可西 | 利落、硬朗 | 黎北、秦戈 |
| 电竞竞技 | 现代姓 | 简短有网感 | 叶临、韩澈 |

#### 7.6.5 热门书角色名黑名单

系统内置一份**banned_names**黑名单，收录以下来源的角色名：

- 起点月票榜Top100作品的主角名和核心配角名
- 各题材经典开山作的角色名（如玄幻的"萧炎""石昊""唐三"）
- 近两年现象级作品角色名

黑名单按题材分级：
- **同题材硬禁止**：同题材的主角名直接禁止（玄幻书不能用"萧炎"）
- **跨题材软警告**：跨题材的核心名给出警告（都市书用"石昊"可以但建议换）
- **配角降级**：配角名同音即可用（主角"萧炎"禁止，配角"萧颜"警告但可用）

用户可随时向黑名单添加/移除条目。

#### 7.6.6 音韵冲突检查

同一书中主要角色名（主角+前10位常驻角色）必须通过以下检查：

| 检查项 | 规则 | 示例 |
|--------|------|------|
| 声母不撞 | 两个主要角色不共享声母 | "沈牧"和"苏婉"声母都是s→警告 |
| 韵尾不撞 | 两个主要角色名韵尾不同 | "江寒"和"苍岚"韵尾都是an→警告 |
| 长度错开 | 同书中不能所有角色都是双字名 | 主角单字"沈牧"，配角双字"裴惊鸿" |
| 首字不撞 | 不同角色的姓/首字不能相同 | "林枫"和"林雪"→警告，读者易混淆 |

#### 7.6.7 用户交互

1. AI一次性为主/副角生成3套候选命名方案
2. 用户可以：整套采用 / 逐个挑选 / 手动修改 / 要求重新生成
3. 用户手动修改后系统仍运行音韵冲突检查
4. 角色名一旦定稿写入characters表，后续章节引用不可再改（需走修史流程）

### 7.7 角色SOUL系统

建书时AI自动为每个角色生成完整的SOUL档案——不仅是"性格描述"，而是可执行的角色灵魂模型，让AI写出的每个角色都有独特的说话方式、行为逻辑和情感模式。

#### 7.7.1 为什么需要SOUL

| 现状 | 问题 |
|------|------|
| 仅靠 `persona_core` 文字描述 | "性格坚韧、重情义"——太笼统，不同作者/AI理解差异巨大 |
| 仅靠 `speech_style` 简短描述 | "说话简洁有力"——无法区分"冷面师兄的简洁"和"老兵的简洁" |
| 无对话示例 | AI无法知道这个角色"具体怎么说话"，千人一面 |
| 无情感模式 | 角色在不同情境下应有不同的反应模式，但现在只有静态描述 |

**SOUL = 角色灵魂模型**：把角色的性格、说话方式、情感反应、行为倾向全部结构化，让AI生成对话和行为时能精确模拟角色。

#### 7.7.4 SOUL生成流程

建书时，系统根据角色原型自动匹配SOUL：

```
角色原型确定（7.5步骤3）
    ↓
SOUL Matcher：从内置SOUL库中为每个角色匹配最合适的SOUL
├── 输入：角色原型 + 题材 + 世界观
├── 匹配逻辑：
│   ├── 1. 优先匹配 genre_compat 完全一致的SOUL
│   ├── 2. 其次匹配核心特征重叠度最高的SOUL
│   └── 3. 若无合适匹配，基于最接近的SOUL自动变体生成
    ↓
SOUL Customizer：根据角色具体设定微调SOUL
├── 调整catchphrases（结合角色名字和世界观特色词）
├── 调整dialogue_examples（结合角色当前身份和处境）
├── 调整behavior细节（结合角色种族/年龄/背景）
└── 调整relationships权重（结合角色间具体关系）
    ↓
用户确认/修改
├── 可替换整个SOUL（从库中重新选择）
├── 可修改SOUL中的任意字段
├── 修改时从内置SOUL库中选择对应子项
└── 也可从空白手动创建自定义SOUL
    ↓
写入characters表的soul_json字段
```

### 7.8 书名生成系统

建书时AI根据用户描述、题材、大纲等已知条件，自动生成有特色的书名供用户选择，核心要求：**有辨识度、有吸引力、一看就想点**。

#### 7.8.1 问题定义

| 现状 | 问题 |
|------|------|
| AI默认书名 | "逆天修仙录""都市重生之xxx"——格式化严重，千篇一律 |
| 用户自行起名 | 要么太直白（"修仙记"），要么太文艺（"浮生若梦"），缺乏网文辨识度 |
| 随便凑词 | 书名与内容脱节，读者被标题骗进来后弃书 |

#### 7.8.2 书名设计原则

1. **题材信号**：书名必须在2秒内让目标读者知道"这是我的菜"（玄幻读者看到"劫"字就想点）
2. **悬念钩子**：书名本身就是一个谜题或承诺，让读者想知道答案
3. **辨识度**：在起点/番茄的书名海洋中不撞名、不俗套
4. **可传播**：读者愿意把书名推荐给别人时不会觉得丢人
5. **可扩展**：书名不把故事锁死，允许剧情超出字面含义

系统对生成的书名执行三层碰撞检查：

| 检查层 | 规则 | 处理 |
|--------|------|------|
| 精确碰撞 | 与起点/番茄等平台已有书名完全相同 | 禁止，必须替换 |
| 高相似碰撞 | 与热门书名相似度>0.7（如"牧道"vs"牧道仙途"） | 警告，建议修改 |
| 品类关键词过载 | 书名中的关键词在同题材Top100中出现频率>30% | 提示，用户决定 |

碰撞数据来源：
- 内置banned_book_titles表（收录各平台月票榜/畅销榜Top200书名）
- 用户手动添加的书名黑名单

#### 7.8.7 用户交互

1. AI一次性生成3-5个候选书名，每个附带风格标签、理由、钩子说明和风险提示
2. 用户可以：直接采用 / 选择后手动修改 / 要求重新生成 / 混合拼接

5. 项目创建初期书名可随时修改；进入正式写作后修改书名需确认（因影响导出和目录）

### 8.1 文风不是"角色口吻"

现有文档的 `speech_style` 只管角色说话方式。但文风是全书级的写作特征：

| 维度 | 说明 | 示例 |
|------|------|------|
| 句式偏好 | 长句/短句/长短交替 | 玄幻偏短句快节奏，悬疑偏中长句营造氛围 |
| 描写密度 | 环境描写占比 | 古风文高描写，都市文低描写 |
| 对话风格 | 对话占比和风格 | 爽文对话多而干脆，细腻文对话少而深 |
| 叙事视角 | 第一人称/第三人称/全知 | 绝大多数网文用第三人称限制视角 |
| 信息释放节奏 | 信息给读者的速度 | 悬疑慢释放，爽文快释放 |
| 情绪渲染方式 | 直接点明/侧面暗示 | 爽文直给("他怒了")，细腻文暗示("指节泛白") |

### 8.4 文风提取Agent（Style Extractor Agent）

用户输入一段文章（自己的旧作、喜欢的作者片段、任何觉得"写得好"的文字），系统逆向提取出完整的文风档案。

#### 为什么必须有文风提取

| 场景 | 没有文风提取 | 有文风提取 |
|------|------------|-----------|
| "我想延续自己上一本的风格" | 手动调metrics参数，靠感觉 | 贴旧作片段，一键提取文风档案 |
| "我想学猫腻的文风" | 无法实现，内置文风里没有 | 导入猫腻的段落，生成"猫腻风"档案 |
| "我觉得这3章写得好，后面保持这个调" | 无从量化"好"在哪 | 提取这3章的文风参数，后续章节自动对齐 |
| "我写的和目标风格差多少" | 无法比较 | 提取自己的片段 vs 目标片段，输出差异报告 |

#### 提取流程

```
用户输入文章片段（500-50000字）
    ↓
文本统计分析（纯计算，不需要LLM）
├── 句长分布：平均句长、最大句长、句长方差
├── 段落长度：平均段长、最大段句数
├── 描写/对话/动作/心理占比
├── 标点密度（句号/逗号/感叹号/省略号比例）
├── 修辞密度（比喻、排比、对偶出现频率）
└── 四字成语密度
    ↓
LLM语义分析（需要LLM）
├── 情绪渲染方式：直接/间接/混合
├── 信息释放节奏：快/中/慢
├── 叙事视角：第一人称/第三人称限制/全知
├── 口语化程度：0-100分
├── 叙事节奏感：紧凑/适中/舒缓
├── 对话风格特征：简洁/冗长/功能型/氛围型
├── 描写偏好：环境为主/人物为主/动作为主
└── 典型过渡词和禁用词识别
    ↓
反AI特征识别
├── 是否包含AI痕迹词（若有，提取后标记为banned_patterns的反面示例）
├── 是否有人类作者特有的"不规范"写法（方言、口语断句、故意冗余）
└── 标注"这个作者绝对不会这样写"的模式
    ↓
合并输出为完整style_profiles档案
    ↓
用户确认/微调 → 保存到全局文风库 → 可被任何项目引用
```

- 支持渐进过渡：设定"从第X章开始，10章内从A文风过渡到B文风"，系统按差异清单逐步调整metrics

## 9. 去AI痕迹引擎

### 9.1 AI痕迹的七种表现

| 编号 | 痕迹类型 | 典型表现 | 示例 |
|------|---------|---------|------|
| A1 | 过度解释 | 把读者当傻子，什么都解释 | "他感到愤怒，因为他觉得这很不公平" |
| A2 | 情绪标签化 | 直接贴情绪词而非展现 | "他感到无比震惊和愤怒" |
| A3 | 句式工整 | 过于对称、排比、四字成语堆 | "山清水秀、鸟语花香、人杰地灵" |
| A4 | 万能过渡词 | AI最爱用的连接词 | "值得注意的是"、"毋庸置疑"、"不可否认" |
| A5 | 描写过度均衡 | 每个场景描写量相似，无轻重 | 每次战斗都是300字环境+300字动作+300字心理 |
| A6 | 人物千人一面 | 所有角色说话方式相同 | 反派也用书面语，小孩也用书面语 |
| A7 | 主动回避冲突 | 不敢写极端情绪和脏话 | 所有人都礼貌克制，没有真正的爆发 |

### 9.2 Voice Filter Agent

在Draft Writer之后、Reader Panel之前插入。

**输入**：章节草稿 + 文风档案 + 角色speech_style

**处理流程**：

```
步骤1：句式扫描 → 检测AI高频句式模式（排比过度、四字成语密度、万能过渡词）
步骤2：情绪扫描 → 检测情绪标签化写法，标注可改为展现式写法的段落
步骤3：描写均衡度扫描 → 检测各场景描写量是否过于均匀
步骤4：角色口吻扫描 → 检测对话是否区分了角色speech_style
步骤5：文风合规扫描 → 检测是否符合当前文风档案的metrics参数
步骤6：生成修正建议 → 输出需要重写的段落和修正方向
步骤7：自动修正 → 对确定性高的AI痕迹直接修正（替换词、删过度解释）
步骤8：标记不确定项 → 对需要创意判断的项标记为risk_flag
```

## 第五部分：大纲体系

## 10b. 大纲体系总览

### 10b.1 为什么大纲是续写的命脉

没有大纲的续写 = AI自由发挥 = 情节漂移。有大纲的续写 = AI在轨道内创作 = 情节连贯。

大纲的核心价值：
- **全书大纲**：锚定终局，防止后文推翻前文
- **分卷总纲**：锚定每卷目标，防止卷间脱节
- **章节大纲**：锚定每章走向，防止章内跑题
- **前5章大纲记忆**：评审时提供近期剧情脉络，防止评审脱离上下文

### 10b.2 四级大纲架构

```
第一级：全书大纲（1份）
├── 总卖点、终局承诺、主题母线、主线走向、核心角色弧光
├── 在建书时自动生成，可修正
└── 全书生命周期内持续生效

第二级：分卷总纲（每卷1份）
├── 卷目标、核心冲突、卷末爆点、阶段成长、卷间衔接
├── 在建书时自动生成，可修正
└── 进入该卷前可调整，进入后冻结核心条款

第三级：事件链大纲（每条事件链1份）
├── 事件链目标、起因、升级点、高潮、结果、回收口
├── 由Arc Planner Agent生成
└── 事件链开始前可调整

第四级：章节大纲（每章1份）
├── 本章目标、推进点、冲突点、情绪走向、尾钩设计
├── 续写时先生成章节大纲，再扩写全章
└── 扩写前可调整，扩写后不可改（只能修改正文）
```

## 第五部分：自动续写系统

## 11. 自动续写架构

### 11.1 续写不是"接着写"

自动续写的核心难题：AI的上下文窗口有限，200万字不可能全部塞进去。续写质量取决于**召回什么、怎么召回、怎么约束生成**。

### 11.2 续写触发

| 触发方式 | 说明 |
|---------|------|
| 手动续写 | 用户点击"续写下一章" |
| 批量续写 | 用户设定范围，连续生成多章（每章仍独立过编译） |
| 事件链续写 | 系统按事件链自动逐章推进，直到事件链完成 |
| 断点续写 | 用户在中途停笔，系统从断点继续 |

### 11.3 续写召回策略（核心）

这是200万字不乱的关键。采用**分层精准召回 + 前章大纲记忆 + RAG语义补充**，绝不全量拼接：

```
第一层：结构性上下文（必带，极小）—— 来自正典层
├── 当前卷目标（1段话）
├── 当前事件链目标（1段话）
└── 本章任务卡（结构化JSON）

第二层：大纲上下文（必带，小）—— 来自大纲层
├── 全书大纲核心条款（终局承诺、主题母线、核心禁忌）
├── 当前分卷总纲
├── 前5章章节大纲 ← 关键：让AI知道近期故事的走向脉络
└── 本章章节大纲（如果已生成）

第三层：状态性上下文（从快照抽取，小）—— 来自快照层
├── 最近章节快照摘要（人物状态、关系变化、位置信息）
├── 最近卷级快照摘要（主线进度、支线状态、世界变化）
└── 活跃伏笔列表（仅未回收的）

第四层：约束性上下文（从正典抽取，小）—— 来自正典层
├── 与本章相关的硬规则（按任务卡关键词匹配）
├── 与本章出场角色相关的核心边界
└── 与本章场景相关的地点/势力规则

第五层：因果性上下文（从账本抽取，小）—— 来自账本层
├── 本章必须推进的事件的前置因果链
├── 本章要回收的伏笔的原始埋设信息
└── 本章涉及的最近关系变化的触发事件

第六层：RAG语义补充（从向量库检索，按需）—— 来自RAG向量层
├── 与本章场景语义相似的历史段落（如"前面类似的密室探索"）
├── 与本章涉及角色相关的历史关键段落（如"林云上次受伤时的描写"）
├── 与本章文风要求匹配的参考段落（如"悬疑场景的典型写法"）
└── 编译器可能需要的跨章证据（如"这个设定之前是怎么表述的"）

第七层：风格性上下文（从文风档案抽取，极小）
├── 当前文风参数
├── 去AI规则
└── 最近3章的尾句（保持行文连贯感）
```

**总量控制**：所有层次合并后不超过8000 token（约12000字），确保生成模型有足够空间创作。RAG层有独立token预算（约1500 token），在总量超限时优先裁剪RAG结果而非结构化数据。

**前5章大纲记忆**是续写连贯性的关键——它让AI不只是知道"现在是什么状态"，还知道"最近5章故事是怎么走的"，从而保证新章与前文的情节脉络衔接。

### 11.4 召回引擎实现

```python
def recall_for_chapter(project_id, chapter_number):
    # 1. 读取任务卡
    task = get_chapter_task(project_id, chapter_number)

    # 2. 获取大纲上下文（大纲层）—— 关键新增
    book_outline = get_book_outline_core(project_id)  # 全书大纲核心条款
    volume_outline = get_volume_outline(project_id, task.volume_id)  # 当前分卷总纲
    prev_5_outlines = get_chapter_outlines(  # 前5章章节大纲
        project_id,
        range(max(1, chapter_number - 5), chapter_number)
    )
    current_outline = get_chapter_outline(project_id, chapter_number)  # 本章大纲（如已生成）

    # 3. 获取最近快照（快照层）
    chapter_snapshot = get_latest_snapshot(project_id, type="chapter")
    volume_snapshot = get_latest_snapshot(project_id, type="volume")

    # 4. 按任务卡关键词匹配正典（正典层）
    canon_refs = match_canon_rules(
        project_id,
        keywords=task.must_progress + task.must_recall + task.must_avoid
    )

    # 5. 按任务卡出场角色匹配状态（账本层）
    character_states = get_active_character_states(
        project_id,
        character_ids=extract_characters(task)
    )

    # 6. 获取活跃伏笔（账本层）
    active_foreshadows = get_active_foreshadows(project_id)

    # 7. 获取因果链（账本层）
    causal_chain = trace_causal_chain(
        project_id,
        event_ids=task.must_progress
    )

    # 8. RAG语义召回（向量层）
    rag_results = rag_semantic_search(
        project_id,
        queries=[
            f"{task.objective} 类似场景",
            *[f"{char} 最近的关键行为" for char in extract_characters(task)],
            *[f["title"] for f in active_foreshadows[:5]],
            f"与{task.objective}匹配的{task.scene_type}场景写法"
        ],
        top_k=5,
        max_tokens=1500,
        exclude_chapter_range=(chapter_number - 3, chapter_number)
    )

    # 9. 组装并裁剪到token限制
    context = assemble_and_trim(
        task, book_outline, volume_outline,
        prev_5_outlines, current_outline,
        chapter_snapshot, volume_snapshot,
        canon_refs, character_states,
        active_foreshadows, causal_chain,
        rag_results,
        max_tokens=8000,
        trim_priority=["rag", "causal_chain", "foreshadows",
                       "character_states", "canon_refs", "snapshot",
                       "prev_outlines", "volume_outline", "task", "book_outline"]
    )

    return context
```

### 11.5 续写流程：先大纲后扩写

续写不是"直接出正文"，而是分两步走：**先生成章节大纲，再根据大纲扩写全章**。这是情节连贯的核心保障。

```
第一步：生成章节大纲
    ↓
用户确认/修改章节大纲 ← 可调整后继续
    ↓
第二步：根据大纲扩写全章
    ↓
Voice Filter去AI → 连续性编译器检查 → 专家评审（携带前5章大纲记忆）→ 全部通过 → 归档+快照
```

**第一步详细流程**：

```
Orchestrator读取当前章节状态
    ↓
读取全书大纲核心条款 + 当前分卷总纲
    ↓
读取前5章章节大纲 ← 关键：了解近期剧情脉络
    ↓
读取最近章节快照 + 活跃伏笔 + 角色状态
    ↓
生成章节任务卡
    ↓
Chapter Outline Agent：生成本章章节大纲
    ↓
输出：本章目标、推进点、冲突、情绪弧、尾钩、伏笔动作
    ↓
用户确认/修改大纲 → 确认后冻结本章大纲
```

**第二步详细流程**：

```
读取本章确认后的章节大纲 + 任务卡
    ↓
Recall Agent：精准召回（含前5章大纲记忆）
    ↓
Draft Writer Agent：根据大纲扩写全章正文
    ↓
Voice Filter去AI + 文风审计
    ↓
连续性编译器（含WordCountChecker）
    ↓ fail → Rewrite Agent → 回到编译器
    ↓ pass
八位专家并行评审（均携带前5章大纲记忆）
    ↓
Review Chair仲裁
    ↓ 通过
Archive Agent：归档 + 写账本 + 生成快照
```

**为什么"先大纲后扩写"比"直接出正文"更连贯**：

| 维度 | 直接出正文 | 先大纲后扩写 |
|------|-----------|------------|
| 情节走向 | AI边写边想，容易跑题 | 大纲已锚定走向，扩写只需填充 |
| 与前文衔接 | 靠召回的碎片信息拼凑 | 前5章大纲提供完整脉络 |
| 用户控制 | 看到正文才发现跑题 | 看到大纲就能纠偏，成本低 |
| 评审效率 | 评审需猜测作者意图 | 大纲已说明意图，评审可对照检查 |

## 第六部分：章节审核与编辑系统

## 12. 多专家评审体系

### 12.1 设计理念：评审必须分科

网文评审不能由一个"全能评委"完成。情节逻辑、人物塑造、节奏爽感、设定一致性、文笔质量、商业潜力——每个维度需要不同的专业知识。混在一起评，要么评不深，要么互相抵消。

**原则**：
- 每个专家只评自己的维度，不越界
- 硬性审核（编译器）和软性评审（专家）分离
- 专家之间结论冲突时，有明确的仲裁规则
- 所有专家输出结构化，可追踪、可对比
- **所有专家评审时必须携带前5章大纲记忆**，不能脱离近期剧情脉络孤立评审

### 12.2 评审架构总览

```
┌─────────────────────────────────────────────────────────────┐
│                    硬性审核层（否决权）                        │

│                                                             │

│                                                             │
│    ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐      │
```

### 12.3 连续性编译器（硬性审核，唯一否决权）

这是唯一有否决权的审核。七项检查全部通过才可进入专家评审流程：

| 检查项 | 输入 | 检查逻辑 | 输出 |
|--------|------|---------|------|
| CanonChecker | 草稿 + 硬规则集合 | 草稿中是否出现与硬规则矛盾的内容 | pass/warning/fail |
| CharacterChecker | 草稿 + 角色边界 + 角色状态 | 角色行为/口吻/能力是否越界 | pass/warning/fail |
| TimelineChecker | 草稿 + 时间线账本 | 事件顺序/移动时间/年龄是否冲突 | pass/warning/fail |
| PowerChecker | 草稿 + 能力表 + 资源账本 | 战力/资源/技能是否跳变 | pass/warning/fail |
| VisibilityChecker | 草稿 + 信息可见性账本 | 角色是否说出其未知信息 | pass/warning/fail |
| ForeshadowChecker | 草稿 + 伏笔账本 | 伏笔回收是否错误/遗漏 | pass/warning/fail |
| WordCountChecker | 草稿 + 项目字数设置 | 章节字数是否在min/max范围内 | pass/warning/fail |

**WordCountChecker规则**：

| 条件 | 结果 | 说明 |
|------|------|------|
| 字数 < min_chapter_words | fail | 章节过短，必须补充内容或合章 |
| 字数 > max_chapter_words × 1.5 | fail | 章节严重超标，必须拆章或大幅精简 |
| 字数 > max_chapter_words 且 ≤ max × 1.5 | warning | 章节偏长，建议精简但不阻止归档 |
| 字数在 min ~ max 范围内 | pass | 正常 |

**失败处理**：
- `fail`：阻止进入专家评审，生成问题列表，推荐回退位置和修复策略
- `warning`：不阻止，但warning自动转发给对应专家（如TimelineChecker的warning转发给设定专家，WordCountChecker的warning转发给节奏专家）

### 12.4 八位评审专家详解

#### 专家1：情节专家（Plot Expert Agent）

**公共输入**：所有专家在评审时都接收以下公共输入，不可省略：

| 公共输入 | 说明 | 为什么必须有 |
|---------|------|------------|
| 本章草稿 | 被评审的章节正文 | 评审对象 |
| 本章章节大纲 | 本章的蓝图 | 对照大纲检查正文是否偏航 |
| 前5章章节大纲 | 近5章的走向脉络 | 没有近期脉络，专家无法判断"这个情节是否突兀"、"这个人物变化是否有铺垫" |
| 本章任务卡 | 本章的约束 | 对照约束检查是否遗漏 |

**角色定位**：像资深责编一样审情节逻辑

| 评审维度 | 检查内容 | 评分标准 |
|---------|---------|---------|
| 因果链完整性 | 本章事件是否有充分的前置铺垫？结果是否合理？ | 每发现一个无因之果扣2分 |
| 推进有效性 | 本章是否实质推进了至少1个核心事件链？ | 零推进 = 不及格 |
| 冲突真实性 | 冲突是否由真实矛盾驱动，而非误会/巧合？ | 巧合驱动扣3分 |
| 悬念设置 | 章末是否制造了有效悬念（尾钩）？ | 无尾钩扣2分 |
| 信息释放 | 本章释放的信息量是否与当前节奏匹配？ | 过多或过少均扣分 |

**输出结构**：

**提示词骨架**：

#### 专家2：人物专家（Character Expert Agent）

**角色定位**：像角色顾问一样审人物塑造

| 评审维度 | 检查内容 | 评分标准 |
|---------|---------|---------|

**提示词骨架**：

### 12.5 评审主席Agent（Review Chair Agent）

**角色定位**：汇总八位专家意见，解决冲突，输出终审报告

**职责**：
1. 汇总8位专家的评分和意见
2. 识别专家之间的意见冲突
3. 按仲裁规则解决冲突
4. 生成优先级排序的修订清单
5. 输出终审结论

**冲突仲裁规则**：

| 冲突类型 | 仲裁原则 | 示例 |
|---------|---------|------|
| 结构合法性 vs 阅读体验 | 合法性优先 | 设定专家说违规，商业专家说好看 → 先修合法性 |
| 文学性 vs 商业性 | 按用户偏好仲裁，默认偏商业 | 文笔专家要改，商业专家说别改 → 用户设定"偏商业"则不改 |
| 不同专家对同一段落的不同意见 | 按严重度排序，高优先处理 | 情节专家说因果链断裂(critical)，节奏专家说水文(low) → 先修因果链 |
| 读者代理团内部分歧 | 取最低分子视角分数 | 老书虫给4分，爽感给8分 → 记录分歧，转给对应专家深入分析 |

**终审结论枚举**：
- `pass`：所有专家≥7分，无must_fix → 可直接定稿

- `needs_human`：专家间严重冲突且无法自动仲裁 → 转人工决定

### 12.6 评审流程与状态流转

```
章节草稿 + Voice Filter版
    ↓
连续性编译器 → fail → 打回重写（不进入专家评审）
    ↓ pass
八位专家并行独立评审
    ↓
评审主席汇总 + 仲裁
    ↓
┌── pass → 可定稿
├── conditional_pass → Rewrite Agent按优先级修复 → 重走编译器+专家评审
├── fail → 建议重新生成初稿
└── needs_human → 转人工编辑决定
```

**并行评审**：八位专家之间无依赖关系，可并行执行，节省时间。

**重评策略**：
- Rewrite修复后只需重评受影响的专家（如只改了水文，只需重评节奏专家和文笔专家）
- 编译器必须每次都重跑
- 修复引入新内容时，所有专家需重评

### 12.7 用户可配置的评审策略

不同用户对评审的严格程度要求不同：

| 策略 | 编译器 | 专家数量 | 通过门槛 | 适用场景 |
|------|--------|---------|---------|---------|
| 严格模式 | 必须 | 全部8位 | 全部≥7分 | 编辑工作室、平台定制 |
| 标准模式 | 必须 | 6位（无商业专家、文风审计简化） | 均分≥7分 | 商业作者日常 |
| 宽松模式 | 必须 | 4位（情节+人物+节奏+读者） | 均分≥6分 | 个人作者初稿快速审核 |
| 自定义 | 必须 | 用户自选 | 用户自定 | 高级用户 |

## 13. 审核状态机

```
task_ready → draft_generated → voice_filtered → compile_check → review_panel
                                                                    ↓
                                        ┌────── compile_failed ←─────┘
                                        ↓
                                    rewrite_required → draft_generated (重走流程)
                                        ↓
                                    review_complete → approved → archived
                                                        ↓
                                              needs_revision (人工标记)
                                                        ↓
                                              rewrite_required
```

**硬性规则**：
- 编译器fail → 必须重写，不可跳过
- 编译器pass + 评审低分 → 可选择重写或定稿（编辑确认）
- 归档后不可自动进入下一章，需人工确认或开启批量模式

## 第八部分：200万字一致性最优解

## 16. 核心难题

200万字以上的网文一致性问题，本质是**信息熵增问题**：随着字数增长，需要维护的事实数量呈指数增长，而AI的上下文窗口是有限的。全量拼接不可行，必须靠系统化的**信息压缩与精准召回**。

## 17. 最优解：四层记忆架构（含RAG向量检索层）

### 17.1 为什么必须有RAG

纯结构化查询（SQLite + FTS关键词匹配）在200万字场景下有四个致命盲区：

| 盲区 | 举例 | FTS能解决？ | RAG能解决？ |
|------|------|-----------|-----------|
| 语义相似但表述不同 | 正文写"左臂骨裂"，任务卡问"主角之前受过类似的伤" | 不能，关键词不匹配 | 能，向量语义相似 |
| 场景级召回 | "前面类似的密室探索场景是怎么写的" | 不能，FTS按词匹配 | 能，段落级向量检索 |
| 隐含信息发现 | 草稿中某句暗示了角色知道秘密，但没明说 | 不能 | 能，语义匹配发现信息越权 |
| 跨章因果链 | "主角为什么信任这个人"需要跨越10章的铺垫证据 | 部分能（需精确知道搜什么词） | 能，语义检索相关段落 |

**结论**：结构化账本是"骨架"，RAG向量库是"血肉"。账本告诉你"林云在第430章受了伤"，RAG能告诉你"他受的是什么伤、当时情境如何、和现在这个场景有什么相似性"。

### 17.2 架构总览

```
┌──────────────────────────────────────────────────────┐
│                   第一层：正典层                        │
│    永久记忆，不可被正文覆盖，全量常驻                    │
│    容量：约2000-5000条规则                              │
│    大小：约5-10万字                                     │
│    存储：SQLite canon_rules表                          │
├──────────────────────────────────────────────────────┤
│                   第二层：快照层                        │
│    周期性状态压缩，替代全量历史扫描                      │
│    按需加载，不常驻                                     │
│    容量：每章约2000字，每卷约5000字                     │
│    存储：SQLite snapshots表                            │
├──────────────────────────────────────────────────────┤
│                   第三层：账本层                        │
│    结构化事实记录，支持精准查询                         │
│    按需召回，按关键词/角色/章节/类型检索                 │
│    容量：数万条记录，但单次召回<100条                   │
│    存储：SQLite 多张账本表                              │
├──────────────────────────────────────────────────────┤
│                   第四层：RAG向量层                     │
│    语义级全文检索，解决"不知道搜什么关键词"的问题         │
│    段落级切片 + 向量索引，支持语义相似度召回              │
│    容量：200万字 ≈ 20000-40000个向量切片               │
│    存储：本地向量数据库                                  │
└──────────────────────────────────────────────────────┘
```

四层的关系不是替代，而是互补：

- **正典层**：回答"规则是什么" → 精确匹配
- **快照层**：回答"当前状态是什么" → 结构化读取
- **账本层**：回答"谁在什么时候做了什么" → 结构化查询
- **RAG向量层**：回答"前面有没有类似的情况" → 语义检索

### 17.2 正典层详解

**定位**：全书的"宪法"，AI生成时必须遵守，正文不可推翻。

**内容**：

| 类别 | 数量估算 | 示例 |
|------|---------|------|
| 世界底层规则 | 50-100条 | "灵气浓度随海拔递减" |
| 力量体系边界 | 50-100条 | "金丹期不可飞行，元婴期才可" |
| 禁忌规则 | 20-50条 | "时间法术只可回溯3秒" |
| 角色核心边界 | 每角色5-10条 | "林云绝不可能背叛师父" |
| 组织核心规则 | 每组织3-5条 | "天机阁不参与世俗争斗" |

**版本管理**：每次修改生成新版本，旧版本保留，修改必须经修史审批。

**常驻策略**：正典总量控制在10万字以内，生成时可全量常驻或按任务卡关键词筛选。

### 17.4 账本层详解

**定位**：结构化的事实数据库，支持精准查询而非全量扫描。

**八类账本**：

|------|---------|---------|
| 世界规则账 | 规则ID、规则类型、适用范围、当前状态 | "查询所有金丹期相关规则" |
| 人物状态账 | 角色ID、章节范围、境界、位置、情绪、已知信息 | "查询林云在第400-438章的状态变化" |
| 关系状态账 | 角色对、关系类型、信任度、冲突度、触发事件 | "查询林云和苏婉的关系变化史" |
| 时间线账 | 章节号、世界日期、相对天数、地点、事件 | "查询第300-400章的时间线" |
| 事件账 | 事件ID、事件类型、因果引用、参与者 | "查询所有导致主角受伤的事件" |
| 伏笔账 | 伏笔ID、埋设章、回收章、成熟条件、重要性 | "查询所有未回收的高重要性伏笔" |
| 信息可见性账 | 信息键、持有者、获取章节、来源事件 | "谁知道主角的真实身份" |
| 资源与代价账 | 物品ID、持有者、消耗规则、冷却规则 | "查询主角当前所有可用技能和剩余次数" |

**账本的增量更新**：每章定稿后，Archive Agent从正文中提取事实变化，增量写入账本。不是重写，是追加。

**账本的精准召回**：Recall Agent根据任务卡的关键词和角色列表，从账本中精准检索需要的条目，而不是全量拼接。

## 18. 一致性保障五道防线

### 18.1 第一道：正典约束（预防）

- 正文生成前，先从正典中筛选相关硬规则注入提示词
- Draft Writer的提示词中明确列出"不可违反"的规则
- 正典是宪法级约束，正文Agent无权修改

### 18.2 第二道：状态快照 + RAG语义记忆（记忆）

- 每章定稿后自动生成快照，压缩当前状态
- 续写时加载最近快照而非全量历史
- 快照确保"500章后仍能准确调取前50章的关键事实"
- **RAG补充**：当快照不够用时（如需要"类似场景"或"角色之前类似的反应"），RAG语义检索补位，不需要精确关键词也能找到相关历史段落

### 18.3 第三道：账本追踪 + RAG语义验证（记录）

- 所有关键事实（人物状态、关系变化、伏笔、时间线）结构化入库
- 事实追溯：任何事实都能找到来源章节
- 冲突定位：新增事实写入前与已有事实做冲突检测
- **RAG补充**：账本是结构化记录，但正文中的隐含信息（如"暗示角色知道秘密"）不一定会被Archive Agent提取入库。RAG能在编译阶段语义搜索全书，发现账本未记录但正文中隐含的语义冲突

### 18.4 第四道：连续性编译（拦截）

- 每章定稿前必须过编译器
- 六项检查（设定/人物/时间线/战力/信息/伏笔）
- 编译器fail = 不可归档
- **RAG增强编译**：编译器不仅检查结构化数据，还通过RAG语义检索验证草稿中的隐含语义是否与前文矛盾。例如草稿中角色说"我知道你的秘密"，RAG检索全书验证该角色是否真的有获取此秘密的事件记录

### 18.5 第五道：修史治理（修复）

- 当必须改史时，先分析影响范围，再选最小修复方案
- **RAG辅助影响分析**：修史时通过语义搜索"提到金丹期不可飞行的所有段落"，而不是依赖人工列出受影响章节
- 三种修复策略：
  - `future_compensation`（后续补偿）：不改旧文，在后续章节补充解释
  - `local_patch`（局部回写）：只修改相关段落的描述
  - `arc_rebuild`（卷级重构）：重建受影响的事件链（最后手段）
- 修史执行后，受影响区间自动进入重验证队列
- 修史执行后，受影响切片从向量库中删除并重新索引

## 19. 装备/道具/技能一致性专项方案

### 19.1 统一物品模型

所有装备、道具、技能、法术统一存储在 `ability_items` 表，用 `item_type` 区分：

| item_type | 示例 | 关键约束 |
|-----------|------|---------|
| weapon | 破天剑 | 持有者、等级要求、特殊效果 |
| consumable | 回灵丹 | 数量、使用条件、效果、使用后减少 |
| skill | 火球术 | 学习条件、消耗、冷却、伤害范围 |
| equipment | 玄铁甲 | 防御值、耐久度、特殊属性 |
| passive | 龙血脉 | 触发条件、效果、限制 |

### 19.2 一致性守恒规则

| 规则 | 检查方式 |
|------|---------|
| 物品唯一性 | 同一物品不可同时被两人持有 |
| 消耗守恒 | 使用消耗品后数量必须减少 |
| 冷却守恒 | 技能冷却期内不可再次使用 |
| 等级守恒 | 使用物品/技能需要满足等级/境界要求 |
| 来源可追溯 | 每件物品的获得方式必须可追溯到获取章节 |
| 升级有前置 | 装备升级/技能突破必须有铺垫事件 |

### 19.3 PowerChecker实现逻辑

```
输入：章节草稿 + ability_items表 + character_states表

步骤1：提取草稿中所有物品/技能使用事件
步骤2：对每个使用事件检查：
  - 持有者是否正确？
  - 数量是否足够（消耗品）？
  - 冷却是否已过？
  - 等级是否满足？
  - 是否有来源记录？
步骤3：对草稿中出现的新物品/技能检查：
  - 是否在正典中有登记？
  - 获得方式是否有铺垫？
步骤4：输出检查结果
```

## 20. 人物一致性专项方案

### 20.1 双层人物模型

```
静态层（characters表）：不随剧情变化的核心属性
├── 核心身份（谁）
├── 核心性格（什么人）
├── 口语风格（怎么说话）
├── 核心边界（绝对不会做什么）
└── 核心动机（为什么做）

动态层（character_states表）：随剧情变化的状态
├── 境界/能力
├── 身体状态
├── 情绪状态
├── 当前目标
├── 所在位置
├── 已知信息
├── 秘密信息
└── 资源/物品
```

### 20.2 人物变化必须绑定触发事件

**硬规则**：任何重要人物状态变化（升级、受伤、情绪转变、关系变化、获取信息）都必须在账本中有对应的触发事件记录。

```
关系变化：信任度40→55，触发事件：苏婉替林云挡了一击
情绪变化：警惕→信任，触发事件：盟友交出了自己的把柄作为诚意
能力变化：金丹初期→金丹中期，触发事件：服用破境丹（第430章获得）
```

编译器检查：如果草稿中人物状态变化没有对应的触发事件，标记为warning或fail。

### 20.3 长期不出场角色的处理

**风险规则**：重要角色超过30章未出场，系统自动触发：
- 状态刷新提醒：该角色在此期间应该有什么变化？
- 伏笔检查：该角色的未回收伏笔是否需要推进？
- 位置追踪：该角色现在应该在哪里？

### 24.3 SOUL Matcher Agent

```text
你是角色SOUL匹配与定制Agent。
你的唯一任务是为每个角色从内置SOUL库中匹配最合适的SOUL模板，并根据角色具体语境进行定制化调整。
```

### 24.4 Book Title Generator Agent

```text
你是书名生成Agent。
你的唯一任务是根据已知条件（描述、题材、大纲、角色名等）为长篇网文生成有特色的书名。

输入条件（可用程度取决于生成时机）：
- 用户描述/logline（必有）
- 题材类型（必有）
- 卷纲/卷目标（可能有）
- 主角名/核心角色名（可能有）
- 全书大纲/终局承诺（可能有）

书名设计原则：
1. 题材信号：2秒内让目标读者知道"这是我的菜"
2. 悬念钩子：书名本身是一个谜题或承诺
3. 辨识度：不与平台热门书撞名，不俗套不格式化
4. 可传播：读者愿意推荐时不会觉得丢人
5. 可扩展：不把故事锁死在字面含义

生成策略：
- 概念型：从logline核心概念提炼（如"牧道"=牧守+证道）
- 人物型：融合主角名与故事（如"沈牧修仙录"→优化→"牧道"）
- 意象型：从世界观核心意象出发（如"九劫"暗示九层成长体系）
- 冲突型：从核心矛盾提炼（如"弃徒之劫"=身份+磨难）
- 承诺型：暗示终局但不剧透（如"一念永恒"暗示选择的力量）

硬约束：
- 禁止使用"之"字结构（"xxx之xxx"格式化严重）
- 禁止使用"天下""无敌""最强"等俗套词作为书名核心
- 禁止直接照搬banned_book_titles中的书名
- 禁止书名超过7个字（移动端显示限制）
- 必须为每个候选书名运行碰撞检查

输出格式：3-5个候选书名，每个包含title/style/rationale/hook/risks/collision_check
```

### 24.6 Draft Writer Agent

```text
你是长篇网文正文生成Agent。
你的唯一任务是根据已确认的章节大纲，在任务卡和正典约束内扩写本章完整正文。

章节大纲（已由用户确认，是本章的蓝图）：
{chapter_outline}

硬约束：
- 必须严格按照章节大纲中的场景顺序和高潮点展开
- 禁止新增未登记硬设定
- 禁止修改角色核心边界
- 禁止让角色说出其未知信息
- 禁止跳过任务卡中的must_progress
- 禁止违反文风档案中的参数限制
- 禁止使用文风档案中的banned_patterns
- 章节字数必须在{min_chapter_words}到{max_chapter_words}之间

- 角色面对冲突必须按SOUL.behavior.conflict_style处理
- SOUL约束优先于文风约束（角色必须像自己，哪怕与叙事节奏不完全一致）

文风约束：
- 句式偏好：{style.metrics}
- 对话占比：{style.dialogue_ratio}
- 描写密度：{style.description_ratio}
- 段落长度：{style.paragraph_max_sentences}句以内

本章必须完成：
{task.must_progress}

本章必须引用：
{task.must_recall}

本章绝对不能：
{task.must_avoid}

扩写原则：
- 大纲是骨架，正文是血肉——允许在场景内增加细节、对话和描写，但不得改变场景走向
- 每个场景必须有足够的篇幅体现其功能，不得一笔带过
- 若大纲中某个场景与正典冲突，停止扩写并报告，不得自行修改

若信息不足，优先保守写法，禁止自行脑补关键规则。
输出正文草稿和风险说明，分开给出。
```

### 24.7 Voice Filter Agent

```text
你是去AI痕迹Agent。
你的任务是扫描章节草稿中的AI写作痕迹并修正。

检查清单：
1. 过渡词黑名单：{banned_transition_words} → 直接删除或改写
2. 情绪标签化：直接写"他感到XXX" → 改为动作/神态/对话展现
3. 句式工整：连续排比、四字成语堆叠 → 打破节奏

6. 口头禅缺失：角色的SOUL.speech.catchphrases在合理场景中未被使用 → 适当加入
7. 禁忌词误用：角色使用了SOUL.speech.taboo_words中的词 → 必须替换
8. 过度解释：把读者当傻子的解释性文字 → 删除或改为含蓄暗示

角色口吻修正规则：
- 每个出场角色的对话必须与其SOUL.speech.tone一致
- 若角色对话句式与SOUL.speech.sentence_pattern冲突，以SOUL为准
- 角色情绪表达方式必须符合SOUL.speech.emotion_expressions
- 参考SOUL.speech.dialogue_examples作为对白风格标杆

修正原则：
- 确定性高的修改（过渡词、情绪标签）直接修正
- 需要创意判断的修改标记为不确定项
- 不要改变情节走向，只改表达方式
- 保持当前文风档案的参数要求
```

### 24.8 Style Extractor Agent


### 24b.5 切换机制

**点击书籍卡片 → 进入该书的工作台**。切换时：

1. 当前书的所有后台任务（编译、评审、健康巡检）暂停或排队
2. 加载目标书的数据库连接、向量库连接、文件路径
3. UI刷新为目标书的项目首页
4. 全局Agent上下文切换：Recall Agent、Draft Writer等只读目标书数据
5. 项目看板（§32b）完整刷新为目标书数据——人物图谱重绘、故事线重建、健康度重算、时间线重载，零残留零泄漏

**切换速度要求**：本地数据库切换 < 200ms，向量库加载 < 500ms，看板首次渲染 < 500ms（详见§32b.7）

**隔离保证**：

| 隔离维度 | 实现方式 | 效果 |
|---------|---------|------|
| 数据库隔离 | 每书独立 `book.db` 文件 | SQL查询天然隔离，不可能跨书 |
| 向量库隔离 | 每书独立ChromaDB collection/目录 | 语义搜索天然隔离 |
| 文件隔离 | 每书独立 `chapters/` 目录 | 正文文件天然隔离 |
| Agent上下文隔离 | 切换书时重新加载project_id | Agent只读当前书数据 |
| 内存隔离 | 切换时清除前书的缓存和快照 | 内存不残留前书数据 |

#### 为什么不用"同一数据库 + project_id隔离"

| 对比项 | 独立DB方案 | 同库project_id方案 |
|--------|-----------|------------------|
| 隔离安全性 | 物理隔离，不可能跨书 | 逻辑隔离，SQL写错就可能跨书 |
| 查询性能 | 单书数据量小，查询快 | 数据量大，需所有查询加project_id条件 |
| 删除/归档 | 删除文件夹即可 | 需逐表删除project_id对应行 |
| 备份/恢复 | 复制文件夹即可 | 需从大库中导出/导入 |
| 并发写入 | 不同书的DB可并发写 | 同库写需锁 |
| 迁移 | 复制文件夹到另一台电脑 | 需从大库中提取 |

**结论**：V1采用独立DB方案，安全、简单、性能好。后续如果需要跨书统计，可以在bookshelf.db中维护聚合视图。

### 24b.10 切换时的上下文清理

```python
def switch_book(target_project_id: str):
    # 1. 暂停当前书的后台任务
    pause_background_tasks(current_project_id)

    # 2. 清除内存中的当前书缓存
    clear_book_cache(current_project_id)

    # 3. 关闭当前书数据库连接
    close_db_connection(current_project_id)

    # 4. 关闭当前书向量库连接
    close_vector_db(current_project_id)

    # 5. 加载目标书数据库
    open_db_connection(target_project_id)

    # 6. 加载目标书向量库
    open_vector_db(target_project_id)

    # 7. 设置全局上下文
    set_current_project(target_project_id)

    # 8. 刷新UI
    refresh_ui(target_project_id)

    # 9. 恢复目标书的后台任务
    resume_background_tasks(target_project_id)
```

**关键保证**：步骤2和4完成之前，步骤5不开始。内存中同一时刻只有一本书的数据。

## 26. RAG向量检索引擎

### 26.1 为什么RAG是200万字长篇的刚需

| 场景 | 纯FTS关键词 | RAG语义检索 |
|------|-----------|-----------|
| "主角之前受过类似的伤" → 正文写的是"左臂骨裂" | 搜不到，表述不同 | 能匹配，语义相似 |
| "前面类似的密室探索场景" | 无法用关键词描述"类似" | 向量距离自然匹配相似场景 |
| "这个设定前面是怎么提的" | 需要精确知道设定名称 | 模糊描述即可找到 |
| 编译器检查"草稿暗示角色知道秘密" | 无法发现隐含语义冲突 | 语义匹配发现信息越权 |
| "前面有人用过这个技能" | 需要精确知道技能名 | 描述技能效果即可匹配 |

**结论**：FTS是"知道搜什么"时的精确工具，RAG是"不知道搜什么"时的语义工具。两者互补，不可替代。

### 26.2 技术选型

| 组件 | V1推荐方案 | 后续扩展 |
|------|-----------|---------|
| 向量数据库 | **ChromaDB**（嵌入式，零部署，Python原生，适合Tauri本地应用） | Milvus（服务端，支持集群） |
| Embedding模型 | 本地：`bge-m3` 或 `nomic-embed-text`（多语言支持好） | 远端API：OpenAI text-embedding-3 |
| 切片策略 | 段落级切片（见下方详解） | — |
| 检索方式 | 余弦相似度 + 元数据过滤 | 混合检索（向量+FTS重排序） |
| 存储位置 | 本地文件 `{project_dir}/vectordb/` | 云端同步 |

### 26.3 切片策略

200万字的正文不能整章存入向量库（粒度太粗，检索精度差），也不能按句子切（粒度太细，丢失上下文）。采用**语义段落级切片**：

```
切片规则：
1. 按自然段落切分，但相邻短段落合并（<200字的段落与下一段合并）
2. 单个切片控制在200-800字
3. 超过800字的段落按语义断点拆分（场景转换、时间跳转、人物切换处）
4. 每个切片携带元数据：project_id, chapter_number, volume_id, arc_id, scene_type, characters
```

**切片元数据结构**：
```json
{
  "chunk_id": "chk_438_07",
  "project_id": "proj_001",
  "chapter_number": 438,
  "volume_id": "vol_04",
  "arc_id": "arc_black_market",
  "chunk_index": 7,
  "content": "林云没说话，指节捏得发白。苏婉递过来一卷绷带，没有多问。...",
  "scene_type": "dialogue_tension",
  "characters": ["林云", "苏婉"],
  "word_count": 342,
  "has_foreshadow": true,
  "foreshadow_ids": ["fs_412"],
  "has_state_change": true,
  "state_changes": ["relationship:林云-苏婉:trust_up"]
}
```

### 26.4 需要向量化的内容

| 内容源 | 切片方式 | 切片量估算（200万字） | 用途 |
|--------|---------|---------------------|------|
| 章节正文定稿 | 段落级切片 | 20000-40000个 | 场景相似性、写法参考、一致性验证 |
| 正典规则 | 每条规则一个切片 | 2000-5000个 | 语义匹配相关规则 |
| 事件账本 | 每个事件节点一个切片 | 3000-8000个 | 因果链语义搜索 |
| 快照摘要 | 每个快照一个切片 | 500-2000个 | 历史状态语义检索 |
| 角色档案 | 每个角色档案一个切片 | 100-1000个 | 角色边界语义匹配 |
| 伏笔记录 | 每条伏笔一个切片 | 500-5000个 | 伏笔语义关联 |
| 爆款参考段落 | 段落级切片 | 5000-20000个 | 写法对标参考 |
| 读者评论 | 每条评论一个切片 | 视量而定 | 语义聚类分析 |

**总量估算**：一个200万字项目约30000-80000个向量切片，本地ChromaDB完全可以支撑。

### 26.5 向量索引维护

| 事件 | 操作 | 时机 |
|------|------|------|
| 章节定稿 | 新增正文切片 + 更新事件/伏笔/状态切片 | 归档时自动 |

**增量更新，不是全量重建**：每次只更新变化的切片，不影响其他向量索引。

### 26.6 检索接口设计

```python
def rag_semantic_search(
    project_id: str,
    queries: list[str],
    top_k: int = 5,
    max_tokens: int = 1500,
    exclude_chapter_range: tuple = None,
    filter_metadata: dict = None,
    # 可选：指定只搜某些内容源
    source_types: list[str] = None,  # ["chapter_text", "canon", "event", "foreshadow"]
    # 可选：混合检索权重
    hybrid_alpha: float = 0.7  # 0=纯FTS, 1=纯向量, 0.7=偏向量的混合
) -> list[RAGResult]:
    """
    RAG语义检索接口

    核心参数说明：
    - queries: 多个查询语句，每个独立检索后合并去重
    - top_k: 每个query返回的最大片段数
    - max_tokens: 总返回量的token上限
    - exclude_chapter_range: 排除太近的章节（避免召回已在上下文窗口内的内容）
    - filter_metadata: 按元数据过滤（如只搜某角色相关段落）
    - source_types: 限制内容源类型
    - hybrid_alpha: 混合检索时向量和FTS的权重
    """
    results = []
    for query in queries:
        # 向量检索
        vector_hits = vector_db.search(
            collection=project_id,
            query_embedding=embed(query),
            top_k=top_k,
            where=build_filter(filter_metadata, exclude_chapter_range, source_types)
        )
        # FTS补充检索（混合模式）
        if hybrid_alpha < 1.0:
            fts_hits = fts_search(project_id, query, top_k=top_k)
            vector_hits = merge_and_rerank(vector_hits, fts_hits, alpha=hybrid_alpha)

        results.extend(vector_hits)

    # 去重（同一片段被多个query命中）
    results = deduplicate(results)

    # 裁剪到token预算
    results = trim_to_token_budget(results, max_tokens)

    return results
```

### 26.7 RAG在六大系统中的调用点

| 系统 | RAG调用场景 | 查询示例 |
|------|-----------|---------|
| **Recall Agent（续写召回）** | 补充结构化数据无法覆盖的语义相似段落 | "与本章密室探索类似的场景" |
| **Continuity Compiler（一致性编译）** | 验证草稿中的隐含语义是否与前文矛盾 | "之前有没有暗示角色知道这个秘密" |
| **Voice Filter（去AI痕迹）** | 检索真人作者的类似场景写法作为参考 | "悬疑紧张场景的典型描写方式" |
| **Retcon Analyst（修史影响分析）** | 语义搜索受影响的所有段落 | "提到金丹期不可飞行的所有段落" |
| **Comment Analyzer（评论分析）** | 将读者评论与正文段落语义匹配 | "读者说'主角太圣母'对应哪些具体段落" |
| **Bestseller Parser（爆款解析）** | 对标作品的写法检索 | "诡秘之主中悬念铺设的典型段落" |

### 26.8 FTS与RAG的协作

FTS和RAG不是二选一，而是各司其职：

| 检索需求 | 用FTS | 用RAG | 说明 |
|---------|-------|-------|------|
| 精确关键词搜索（"金丹期"） | 是 | — | 知道搜什么时，FTS更快更准 |
| 语义相似搜索（"类似受伤的场景"） | — | 是 | 不知道搜什么关键词时，RAG更准 |
| 角色名搜索 | 是 | — | 精确名称，FTS足够 |
| "前面有没有提过这个概念" | 补充 | 主 | 概念可能用不同词表达，RAG优先 |
| 混合搜索 | 混合 | 混合 | 重要场景同时用两种方式，合并重排序 |

建议建立FTS表：
- `chapters_fts`：章节正文全文检索（精确关键词）
- `canon_rules_fts`：正典规则全文检索
- `event_nodes_fts`：事件账本全文检索
- `reader_comments_fts`：读者评论全文检索

```
Orchestrator读取当前章节状态
    ↓
Task Card Agent：生成本章任务卡（约束：必须做什么/不能做什么）
    ↓
Chapter Outline Agent：生成本章大纲（蓝图：故事怎么走、场景怎么排）
    ├── 输入：任务卡 + 前5章大纲 + 卷纲 + 事件链 + 正典 + 角色状态
    └── 输出：章节大纲JSON（scenes, climax_point, ending_hook, foreshadow_actions）
    ↓
用户确认大纲（可修改、可要求重新生成）
    ↓
Recall Agent：精准召回本章所需上下文（分层召回，≤8000 token）
    ↓
Draft Writer Agent：根据已确认大纲扩写全章正文（遵守正典+文风约束）
    ↓
Voice Filter Agent：去AI痕迹修正 + 文风合规报告
    ↓
Continuity Compiler：运行七项规则检查（含字数检查）
    ↓ fail → Rewrite Agent定向修复 → 回到编译器
    ↓ pass
八位专家并行评审（所有专家携带前5章大纲记忆）
├── 情节专家：因果链 + 推进 + 悬念 + 与前5章衔接
├── 人物专家：行为一致性 + 对话辨识度 + 弧光 + 与前5章延续性
├── 节奏专家：爽点分布 + 张力曲线 + 水文检测 + 与前5章节奏连贯
├── 设定专家：体系守恒 + 设定膨胀 + 地理逻辑 + 与前5章设定一致
├── 文笔专家：句式 + 描写 + AI味残留 + 与前5章笔触连贯
├── 商业专家：爽点密度 + 代入感 + 追读驱动力 + 与前5章动力衔接
├── 读者代理团：四子视角独立评审
└── 文风审计：文风参数合规 + 禁用模式
    ↓
Review Chair Agent：汇总 + 仲裁 + 输出终审报告
    ↓
├── pass → 可定稿
├── conditional_pass → Rewrite Agent按优先级修复 → 重走编译器+受影响专家
├── fail → 建议重新生成初稿
└── needs_human → 转人工编辑决定
    ↓ 通过
Archive Agent：归档 + 写账本 + 生成快照
```

### 32b.7 切换书籍时看板数据同步

切换书籍是看板的核心交互场景。所有看板数据必须随切换完整刷新，零残留。

#### 数据同步流程

```
用户点击书架中的另一本书
    ↓
1. 当前书看板暂停所有实时更新
    ↓
2. 释放当前书的DB连接 + 向量库连接
    ↓
3. 加载目标书的DB连接 + 向量库连接
    ↓
4. 从目标书数据库重算看板所需数据：
   ├── 总览页: projects + volumes + chapters 聚合查询
   ├── 人物图谱: characters + character_states + relationship_states 图计算
   ├── 故事线: volumes + arcs + foreshadow_items 树状结构
   ├── 健康度: compiler_reports + 最近快照 指标计算
   └── 时间线: timeline_nodes + locations 时间序列
    ↓
5. 渲染目标书看板
    ↓
6. 恢复目标书的实时更新推送
```

#### 性能要求

| 看板页 | 切换后首次渲染 | 增量更新 |
|--------|--------------|---------|
| 总览 | < 300ms | < 100ms（章节定稿后） |
| 人物图谱 | < 500ms（100角色内） | < 200ms（关系变更后） |
| 故事线 | < 400ms | < 150ms（大纲变更后） |
| 健康度 | < 300ms | < 200ms（章节定稿后） |
| 时间线 | < 400ms | < 150ms（章节定稿后） |

#### 缓存策略

- 每个书的看板数据在内存中缓存最近一次计算结果
- 切换回已缓存的书时，先渲染缓存版本（< 50ms），再后台刷新
- 缓存失效条件：章节定稿、正典变更、修史执行、角色状态变更

#### 数据隔离保证

- 图谱计算只读当前书的`characters`和`relationship_states`
- 故事线只读当前书的`volumes`、`arcs`、`foreshadow_items`
- 健康度只统计当前书的`compiler_reports`
- 时间线只读当前书的`timeline_nodes`
- 切换后清空所有看板组件的本地状态，防止A书数据残留在B书页面

### 32b.8 看板数据查询接口

看板不引入新数据表，所有数据从已有表聚合查询：

| 看板组件 | 数据来源 | 查询方式 |
|---------|---------|---------|
| 进度概览 | projects + volumes + chapters | `SELECT COUNT(*), SUM(word_count) FROM chapters WHERE project_id=? AND status='archived'` |
| 待办事项 | compiler_reports(status=fail) + retcon_requests(status=pending) + foreshadow_items(超期) + character_states(失联) | 多表UNION查询 |
| 人物图谱 | characters + relationship_states | 全量加载角色+关系，前端力导向布局 |
| 角色状态 | character_states + foreshadow_items | `SELECT * FROM character_states WHERE character_id=? ORDER BY chapter_from DESC LIMIT 1` |
| 故事线全局图 | volumes + arcs + foreshadow_items | 树状递归查询 |
| 卷级事件链 | arcs + chapter_tasks + chapters | `SELECT * FROM arcs WHERE volume_id=?` |
| 伏笔全览 | foreshadow_items | `SELECT * FROM foreshadow_items WHERE project_id=? ORDER BY importance DESC` |
| 健康度指标 | compiler_reports + chapters + canon_rules + foreshadow_items | 聚合计算（见§32） |
| 时间线 | timeline_nodes + characters + locations | `SELECT * FROM timeline_nodes WHERE project_id=? ORDER BY relative_day` |
| 写作统计 | chapters(created_at) | 按日期分组聚合 |

## 33. 风险预警规则

| 规则 | 触发条件 | 建议操作 |
|------|---------|---------|
| 角色失联 | 重要角色>30章未出场 | 状态刷新提醒 + 伏笔检查 |
| 设定膨胀 | 近50章新增设定>15%原设定 | 触发设定膨胀警报，建议合并或冻结 |
| 伏笔过期 | 高重要性伏笔>预期回收卷3卷仍未回收 | 触发延期回收提醒 |
| 战力通胀 | 主角近20章连续升级无代价 | 触发战力检查，要求增加代价铺垫 |
| 时间线模糊 | 连续5章无时间线记录 | 触发时间线补录提醒 |
| AI味回升 | 连续3章AI残留率>10% | 触发文风参数调整建议 |
| 章节字数异常 | 单章字数 < min_chapter_words 或 > max_chapter_words | 提示章节过短/过长，需补充或精简 |
| 字数持续下降 | 连续10章字数递减 | 可能注水减少或节奏变化 |
| 字数持续上升 | 连续10章字数递增 | 可能描写膨胀 |
| 描写占比偏离 | 实际描写占比偏离文风参数>30% | 提示描写过多或过少 |
| 写作速度骤降 | 近7天日均字数 < 近30天日均 × 50% | 提示写作效率下降 |

## 35. 总结

NovelOS Longform 的核心设计思想：

1. **一键启动**：题材模板 + 文风档案 + 自动预填充，让用户从一句话到完整项目只需30秒
2. **去AI味**：Voice Filter Agent + 去AI规则库 + 文风参数约束，三管齐下
3. **学爆款**：爆款解析Agent + 模式沉淀库 + 对标写作，让AI学会真人作者的写法
4. **自动续写**：分层精准召回（≤8000 token），绝不全量拼接，续写质量不随字数增长衰减
5. **多专家评审**：8位专业Agent并行评审 + Review Chair仲裁，情节/人物/节奏/设定/文笔/商业/读者/文风各有专攻
6. **读者闭环**：评论分析 → 修订建议 → 编辑确认 → 执行修订 → 重新审核，完整闭环
7. **多书并行**：每书独立数据库+独立向量库+独立文件目录，物理隔离零泄漏，全局共享题材/文风/模式库
8. **200万字不乱**：四层记忆架构（正典层 + 快照层 + 账本层 + RAG向量层）+ 五道防线（正典约束 → 快照+RAG语义记忆 → 账本+RAG语义验证 → RAG增强编译 → 修史治理），这是超长篇一致性的最优解

这不是一个"加大上下文的AI写作器"，而是一套多书并行、从用户一句话到百万字成书的完整内容操作系统。

| P2 | **互动式剧情探索** | Subtxt(Emergent Story Fields) | "扮演你的角色做选择"式探索，帮助作者发现剧情可能性 | Phase 5 |
| P2 | **30+剧情结构模板** | Plottr | 当前只有10种题材模板，可扩展为每题材3-5种剧情结构变体 | Phase 4 |
| P2 | **系列/续作管理** | NovelCrafter(跨书Codex) | 续作共享世界观和角色时，可复用前作的正典和SOUL | Phase 5 |

### 36.4 中国网文工具市场空白

当前中文AI写作工具（笔灵AI、彩云小梦、秘塔写作猫、火山写作）全部是**通用写作工具+小说分类**的模式，无一具备：
- 角色追踪和一致性管理
- 世界观数据库
- 时间线管理
- 编译式一致性检查
- 修史治理

NovelOS在中文网文AI写作赛道**无直接竞品**，核心壁垒在于"长篇一致性工程化"能力。

### 38.5 总结

NovelOS Longform 在以下维度已**超越所有竞品**：
- 长篇一致性工程化（五道防线+编译式检查，竞品无此能力）
- 角色口吻控制（SOUL结构化档案+30内置模板，竞品仅靠微调或文本描述）
- 修史治理（影响分析+最小修复+审批流，独创能力）
- 读者反馈闭环（竞品无此能力）
- 网文题材深度适配（10种题材+节奏参数+爽点曲线，竞品均为通用结构）

需要补全的核心缺口按优先级：
1. **已有稿件导入**（P0，阻塞现有作者采纳）
2. **Archive Agent规范**（P0，全系统最复杂但零规范）
3. **Agent失败/重试/Token预算**（P0，生产环境必需）
4. **通知系统**（P0，风险预警不推送等于没有）
5. **自定义日历**（P1，玄幻/历史题材刚需）
6. **BYO-AI架构**（P1，避免供应商锁定）
7. **导出格式规范**（P0，写作工具的必备出口）

补全以上7项后，NovelOS将成为功能完整度显著领先的长篇AI写作系统。
